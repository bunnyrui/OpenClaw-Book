# G2.1 Skills 安装与使用
> **生成模型**：Claude Opus 4.6 (anthropic/claude-opus-4-6)
> **Token 消耗**：输入 ~180k tokens，输出 ~7k tokens（本节）

---

Skills 是 OpenClaw 给 AI Agent 加能力的主要方式。你可以把它理解成一组可热插拔的能力模块：默认不用全部加载，需要时再装、再启用、再让模型去读对应的 `SKILL.md`。这样做的好处很直接：主系统保持轻，能力又能按场景扩展。

## G2.1.1 什么是 Skill

一个 **技能**（Skill）本质上就是一个目录，里面至少有一个 `SKILL.md`。这个文件会告诉模型三件事：这个 Skill 适合解决什么问题、应该调用哪些工具、调用时要遵守什么约束。

> **衍生解释**：这里的“热插拔”不是内核驱动那种热插拔，而是说你不需要改 OpenClaw 主程序代码，只要把 Skill 放到约定目录、满足依赖，它就能在新的会话里被发现并使用。

## G2.1.2 Skill 的三种来源

OpenClaw 里常见的 Skill 有三类：

1. **内置技能**（Bundled Skills）：随 OpenClaw 一起发布，当前仓库里有 52+ 个，比如 `coding-agent`、`github`、`weather`、`canvas`。
2. **托管技能**（Managed Skills）：放在 `~/.openclaw/skills`，通常用于跨项目复用，或者通过 ClawHub 管理。
3. **工作区技能**（Workspace Skills）：放在 `<workspace>/skills`，只对当前项目生效，适合团队私有约定。

优先级是：`<workspace>/skills` > `~/.openclaw/skills` > 内置 Skills。

## G2.1.3 怎么查看和发现 Skills

先看本机已经有哪些 Skill：

```bash
openclaw skills list
openclaw skills list --eligible
openclaw skills info coding-agent
openclaw skills check
```

其中 `--eligible` 会只显示当前环境下“可直接使用”的 Skill，比如已经装好了 `gh`、`curl`、`claude` 之类依赖。

如果你想找新 Skill，实际工作流一般是去 ClawHub：

```bash
clawhub search "github"
clawhub search "weather"
clawhub list
```

> **提示**：很多人会把“安装 Skill”口头说成 `openclaw skills install ...`，但当前版本里 `openclaw skills` 主要负责查看和检查；真正从 ClawHub 拉取 Skill，通常用的是 `clawhub install <slug>`。

## G2.1.4 安装 Skill 的实际命令

常见安装方式如下：

```bash
# 先装 ClawHub CLI
npm i -g clawhub

# 安装到当前工作区的 ./skills
clawhub install coding-agent
clawhub install github
clawhub install weather
clawhub install discord
clawhub install canvas
```

安装后，再回到 OpenClaw 里检查：

```bash
openclaw skills list --eligible
openclaw skills info github
```

如果你想共享给所有 Agent，也可以把 Skill 放到：

```bash
~/.openclaw/skills/<skill-name>/SKILL.md
```

## G2.1.5 示例一：`coding-agent`

`coding-agent` 是非常实用的一个 Skill。它的核心作用不是“自己写代码”，而是教 OpenClaw 何时调用外部编程代理，例如 Codex、Claude Code、OpenCode 或 Pi，然后配合 **Bash 工具**（Bash Tool）和文件工具完成改代码、跑测试、看日志这些任务。

先确认它是否可用：

```bash
openclaw skills info coding-agent
```

如果状态里提示缺少 `claude`、`codex`、`opencode` 或 `pi`，就先装对应 CLI。随后你可以直接对 AI 说：

```text
请在当前项目里添加一个 REST API，并跑一遍测试。
```

有了 `coding-agent` 之后，模型会倾向于：先读 Skill 说明，再调用 Bash、Read、Write、Edit 之类工具，而不是只给你一段“纸上谈兵”的代码建议。

## G2.1.6 示例二：`github`

`github` Skill 依赖 `gh` CLI，适合把“创建 Issue、查看 PR、检查 CI”这些动作交给 Agent。

```bash
openclaw skills info github
gh auth login
gh auth status
```

然后你可以直接说：

```text
帮我在 owner/repo 创建一个 bug issue，标题是“登录页在 Safari 崩溃”。
```

或者：

```text
查看这个仓库最近 5 个 PR 的 CI 状态，并总结哪个卡住了。
```

## G2.1.7 示例三：`weather`

`weather` Skill 很轻，通常只要求系统里有 `curl`。它适合问即时天气和短期预报：

```bash
openclaw skills info weather
```

你可以这样提问：

```text
帮我查一下杭州未来三天的天气，顺便告诉我周末适不适合骑行。
```

这类 Skill 的价值在于：模型不必“假装知道天气”，而是能按 Skill 说明去请求真实数据源。

## G2.1.8 在 `openclaw.json` 里配置 Skill

单个 Skill 的开关、环境变量、私有配置，通常写在 `skills.entries`：

```json5
{
  skills: {
    entries: {
      github: {
        enabled: true,
      },
      weather: {
        enabled: true,
      },
      "coding-agent": {
        enabled: true,
      },
      "my-company-skill": {
        enabled: true,
        apiKey: "$MY_COMPANY_API_KEY",
        env: {
          MY_COMPANY_API_KEY: "$MY_COMPANY_API_KEY",
        },
        config: {
          endpoint: "https://api.example.com",
          team: "infra",
        },
      },
    },
  },
}
```

`enabled: false` 可以临时关掉某个 Skill；`env` 和 `apiKey` 则常用于给 Skill 注入密钥。

## G2.1.9 写一个自己的 Skill

最简单的自定义 Skill，可以先从工作区开始：

```bash
mkdir -p ./skills/release-helper
```

然后创建 `./skills/release-helper/SKILL.md`：

```markdown
---
name: release-helper
description: 帮助发布 npm 包，整理变更日志，并调用现有 shell 工具。
---

# Release Helper

当用户要求发布版本时：

1. 先读取 `package.json` 和 `CHANGELOG.md`。
2. 再使用现有的 Bash 工具执行 `npm version`、`npm publish`、`git tag`。
3. 如果需要斜杠命令，可继续补充 `command-dispatch`、`command-tool` 等定义。
```

这个例子里，你并没有写新的运行时代码，而是在 `SKILL.md` 里给模型补了一套稳定操作说明。对于很多团队内部流程，这已经够用了。

## 本节小结

Skills 可以理解成 OpenClaw 的能力插件层：内置 Skill 解决常见任务，ClawHub 负责分发第三方 Skill，工作区 Skill 适合项目私有流程。日常使用时，先用 `openclaw skills list` 看状态，再用 `clawhub install` 安装需要的能力，最后通过 `skills.entries` 做细粒度配置。Skill 系统的架构详见第 21 章。
