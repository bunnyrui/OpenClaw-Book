# G2.3 中国平台集成
> **生成模型**：Claude Opus 4.6 (anthropic/claude-opus-4-6)
> **Token 消耗**：输入 ~180k tokens，输出 ~7k tokens（本节）

---

如果你在北美或欧洲做个人助手，第一反应往往是 Telegram、Discord、Slack。但对国内用户来说，真正高频的平台通常是飞书、钉钉、企业微信。OpenClaw 的通道扩展思路，恰好适合接这类平台：核心 Gateway 不动，把接入能力放进扩展层。

## G2.3.1 为什么这件事重要

中国团队的沟通工具有一个很明显的特点：办公沟通、审批流、机器人能力、企业账号体系往往绑得很紧。也就是说，AI 助手如果只会接 Telegram/Discord，在很多公司里其实落不了地。

> **衍生解释**：这里的“平台集成”不是简单地“能发消息”就算完成，而是要让 OpenClaw 能接收入站事件、回发消息、管理会话，必要时还要适配企业认证和网络环境。

## G2.3.2 通用安装模式：插件与扩展

OpenClaw 的通道扩展通常按 **插件**（Plugin）交付，本质上可以看作 npm 包，安装命令统一是：

```bash
openclaw plugins install <path-or-spec>
openclaw plugins list
openclaw plugins doctor
```

比如从 npm 安装：

```bash
openclaw plugins install @openclaw/feishu
```

如果某个扩展在你的源码仓库里提供了本地目录，也可以直接从本地路径安装：

```bash
openclaw plugins install ./extensions/<extension-name>
```

> **提示**：并不是每个中国平台扩展都已经作为官方包发布。判断是否“现成可用”，最稳妥的方式还是看当前版本文档、`extensions/` 目录，或者插件市场。

## G2.3.3 飞书（Feishu/Lark）接入

飞书是当前中国平台里资料最完整、接入体验也比较成熟的一类。常见流程如下：

```bash
# 某些版本已内置，老版本或精简安装可手动补装
openclaw plugins install @openclaw/feishu

# 然后按交互方式添加渠道
openclaw channels add
openclaw gateway status
openclaw logs --follow
```

配置文件写法大致如下：

```json5
{
  channels: {
    feishu: {
      enabled: true,
      dmPolicy: "pairing",
      accounts: {
        main: {
          appId: "cli_xxx",
          appSecret: "$FEISHU_APP_SECRET",
          botName: "OpenClaw 助手",
        },
      },
    },
  },
}
```

如果你走 Webhook 模式，还要在飞书开放平台配置 **事件订阅地址**（Event Subscription URL）和 `verificationToken`。而在长连接模式下，则更像是由 OpenClaw 主动连过去接收事件。

Lark 国际版也能沿用同一套思路，只是平台域名和部分配置项不同。

## G2.3.4 钉钉（DingTalk）

钉钉在企业里也非常常见，但在 OpenClaw 生态里，现成扩展的完备度通常不如飞书。实际落地时有两条路线：

1. 如果当前版本存在官方或社区扩展，就按插件方式安装，例如：`openclaw plugins install <dingtalk-extension>`。
2. 如果没有现成扩展，就先用 **Webhook 集成**（Webhook Integration）做一个轻量入口。

Webhook 方案常见做法是让钉钉机器人把消息推给你自己的中间层，再由中间层转发到 OpenClaw 的 Webhook 或 Gateway 接口。这样虽然多了一层，但实现成本低，适合先做 PoC。

```bash
openclaw plugins install <dingtalk-extension>
openclaw plugins doctor
```

> **提示**：书里这里特意写成占位形式，是因为扩展是否存在、包名是什么，要以你手头版本为准。实战里千万别只凭博客截图硬配。

## G2.3.5 企业微信（WeCom）

企业微信的接法和钉钉很像：如果有现成扩展，就按插件方式装；如果没有，就采用企业机器人 + Webhook 中转。

关键点通常有三个：

1. 企业内部应用或群机器人凭证怎么拿；
2. 入站消息怎么转成 OpenClaw 能理解的事件；
3. 出站消息怎么映射回企业微信的发送接口。

从架构上说，这和其他 Channel Extension 没有本质区别，只是平台 API、签名校验、消息格式不同。

## G2.3.6 中国部署时的现实问题

中国平台接入除了“功能能不能跑”，还要考虑网络和运维。

第一是 API 可达性。某些国际模型提供者在国内网络环境下会不稳定，可能需要代理；如果你希望体验稳一点，主模型通常更适合先换成 DeepSeek、Qwen 之类国产提供者。

第二是延迟。把 Gateway 部署在阿里云、腾讯云这类国内 VPS 上，实际交互延迟通常会比海外机器更好。

第三是运维复杂度。很多团队最后都会选择 Docker：

```bash
docker compose up -d
docker compose logs -f
```

这样做的好处是升级、迁移、环境复现都更简单。

## G2.3.7 微信集成要特别谨慎

很多人最关心的是微信，但微信的开放接口限制比企业协同平台严格得多。个人号自动化、消息监听、双向收发，在合规和技术层面都比飞书、钉钉、企业微信更敏感。

所以更现实的建议是：

1. 企业场景优先做企业微信；
2. 内部办公优先做飞书或钉钉；
3. 微信个人号不要当成首发主通道。

> **衍生解释**：这里的限制不只是“有没有 API”，还包括风控、账号稳定性、审核规则和后续维护成本。很多项目不是做不出来，而是长期不可维护。

## G2.3.8 一个实战上的推荐顺序

如果你现在就要在中国团队里落地 OpenClaw，可以按这个顺序推进：

```bash
# 1. 先把模型换成低延迟方案
openclaw models set deepseek/deepseek-chat

# 2. 装可用的企业平台扩展
openclaw plugins install @openclaw/feishu

# 3. 启动网关并看日志
openclaw gateway status
openclaw logs --follow
```

先打通“能聊天、能收发消息、能切模型”这条最短路径，再考虑审批流、企业知识库、内部系统联动这些增强能力。

## 本节小结

中国平台集成的关键，不是单纯把 OpenClaw 接到某个聊天工具，而是把插件机制、模型选择、部署环境一起考虑。飞书通常是最成熟的起点；钉钉、企业微信更适合根据现成扩展或 Webhook 方案逐步推进；微信则要格外注意接口限制和长期维护成本。通道扩展机制详见第 13 章，部署方案详见第 29 章。
