> **Token 消耗**：输入 ~180k tokens，输出 ~7k tokens（本节）

---

前两节解决了“装上”和“接上”的问题，这一节开始碰 OpenClaw 真正有味道的部分：它不只是回答你，还能替你做事。

## G1.3.1 工具到底是什么

在 OpenClaw 里，**工具**（Tool）可以理解成“AI 被允许调用的外部能力”。常见能力包括：

- 执行 bash 命令
- 浏览网页并提取信息
- 读写文件
- 调用浏览器、Cron、节点等系统能力

如果把模型比作“大脑”，那工具更像“手和脚”。没有工具，Agent 只能建议；有了工具，它才可能真的执行。

> **衍生解释 — Tool Calling**
>
> Tool Calling 不是让模型“偷偷运行程序”，而是让模型先输出一个结构化调用请求，再由 OpenClaw 决定是否真的执行。也就是说，模型负责提出“我想用哪个工具做什么”，执行权仍然在宿主系统和策略层手里。

## G1.3.2 例子一：让 Agent 用 bash 查天气

先保证 Gateway 已经启动，然后直接给 Agent 下一个具体任务：

```bash
openclaw agent --message "请用 bash 执行 curl 'wttr.in/Beijing?lang=zh-cn'，然后用中文总结今天北京天气。"
```

如果你的工具策略允许 bash，这次对话通常会经历下面几步：

1. Agent 判断“需要真实天气信息，不能靠猜”
2. Agent 调用 bash 工具执行 `curl wttr.in/...`
3. OpenClaw 获取命令输出
4. Agent 再把结果整理成自然语言回复你

你也可以把城市换成自己的真实场景：

```bash
openclaw agent --message "请用 bash 执行 curl 'wttr.in/Shanghai?lang=zh-cn'，告诉我今天是否适合骑车上班。"
```

## G1.3.3 例子二：让 Agent 打开网页并总结

OpenClaw 还可以借助浏览器能力访问网页，再返回摘要。一个实用例子是直接总结官方文档页面：

```bash
openclaw agent --message "请打开 https://docs.openclaw.ai/start/quickstart ，阅读后用 5 条总结 OpenClaw 的快速开始流程。"
```

如果浏览器工具可用，Agent 不需要你先复制粘贴网页内容。它会自己去拿页面，再把重点压缩给你。

```typescript
const browseTask = {
  open: "访问目标网页",
  inspect: "提取标题、段落、关键命令",
  summarize: "根据你的要求重新组织输出",
};
```

这类能力很适合做“读文档、提炼重点”的重复劳动。

## G1.3.4 定时任务：让 OpenClaw 按时干活

如果你希望 OpenClaw 每天、每小时、每周自动做事，就要用 **Cron**（Cron）调度。

很多人第一次都会想直接用自然语言描述任务，比如“每天早上 9 点给我发送天气预报”。这个想法没问题，但当前 CLI 需要显式提供调度表达式和消息内容。通常做法是先让 Agent 帮你把自然语言变成 Cron 表达式，再正式创建任务。

你可以先这样问：

```bash
openclaw agent --message "把‘每天早上9点给我发送天气预报’转换成 cron 表达式，并顺手解释每一列的含义。"
```

等 Agent 告诉你结果是 `0 9 * * *` 之后，再创建真正的任务。

比如“每天早上 9 点发送天气预报”，可以写成：

```bash
openclaw cron add \
  --name "Morning Weather" \
  --cron "0 9 * * *" \
  --session isolated \
  --message "查询今天杭州天气，并给我一段通勤建议。" \
  --announce \
  --channel telegram \
  --to "123456789"
```

创建后可以马上检查：

```bash
openclaw cron list
openclaw cron status
```

## G1.3.5 `openclaw.json` 里的 Cron 任务长什么样

虽然平时更建议用 CLI 添加，但理解配置结构很有帮助：

```json5
{
  cron: {
    jobs: [
      {
        name: "Morning Weather",
        enabled: true,
        schedule: {
          kind: "cron",
          expr: "0 9 * * *",
          tz: "Asia/Shanghai",
        },
        sessionTarget: "isolated",
        payload: {
          kind: "agentTurn",
          message: "查询今天杭州天气，并给我一段通勤建议。",
        },
      },
    ],
  },
}
```

你可以把一个 Cron 任务理解成三部分：

- 什么时候运行
- 运行在哪个会话里
- 到点以后让 Agent 做什么

## G1.3.6 Webhook：不是按时间，而是按事件触发

Cron 是“时间到了就运行”，Webhook 则是“外部事件来了就运行”。比如：

- 某个表单有新提交
- 某个服务推送了一条通知
- 某个自动化平台向你的 Gateway 发了一个 HTTP 请求

你现在只要先记住这个区别：

- Cron：时间驱动
- Webhook：事件驱动

后面你把这两种方式结合起来，就能做出很像个人自动化助理的东西。工具系统详见第 14-16 章，Cron 系统详见第 18 章。

> **提示**
>
> 学习这一节时，建议你至少自己跑一遍“bash 查天气”和“创建一个定时任务”。只看例子很容易觉得自己懂了，真到实操时最容易卡在命令参数和投递目标上。

## 本节小结

1. 工具是 OpenClaw 把“会说”变成“会做”的关键，常见能力包括 bash、网页浏览和文件处理。
2. 让 Agent 用 bash 查天气，是理解工具调用链路的最好起点。
3. 网页总结场景体现了 OpenClaw 的另一个实用方向：替你读页面、提重点、缩短信息摄入时间。
4. 当前 `cron add` 的实际 CLI 需要显式给出 `--cron`、`--message` 等参数，自然语言通常先交给 Agent 帮你转换。
5. Cron 是时间驱动，Webhook 是事件驱动；这两套机制会在后面的自动化章节里反复出现。
