# G1.1 安装与首次启动

> **生成模型**：Claude Opus 4.6 (anthropic/claude-opus-4-6)
> **Token 消耗**：输入 ~180k tokens，输出 ~7k tokens（本节）

---

很多人第一次接触 OpenClaw，会把它想成“一个能在终端里聊天的 AI 工具”。这样理解不算错，但还不够完整。OpenClaw 更像一个运行在你自己设备上的 AI 助手底座：它有自己的 Gateway 网关、会话、模型配置、通道适配器和工具系统。本章先不碰源码，我们先把它装起来、跑起来、发出第一条消息。

## G1.1.1 先准备好运行环境

安装前，你至少需要下面三样东西：

- **Node.js**（Node.js）`22+`
- 一台可长期运行 OpenClaw 的电脑：macOS、Linux，或者 Windows + WSL
- 能联网访问模型服务的环境

你可以先检查 Node 版本：

```bash
node --version
npm --version
```

如果输出的 Node 主版本小于 22，先升级再继续。OpenClaw 的很多运行时能力依赖较新的 Node 特性，版本太老时常见问题不是“不能启动”，而是看起来能装、实际跑不稳。

> **提示**
>
> 如果你在 Windows 上学习本书，优先使用 WSL。这样你在终端命令、路径习惯、权限模型上会更接近 Linux/macOS，后面做 Cron、日志排查、工具调用时也更顺手。

## G1.1.2 全局安装 OpenClaw

最直接的安装方式是全局安装 CLI：

```bash
npm install -g openclaw
```

安装完成后，先确认命令已经进入 PATH：

```bash
openclaw --help
openclaw doctor
```

`openclaw doctor` 是很实用的健康检查命令。它会帮你检查环境、配置、依赖和常见错误。后面如果你遇到“命令能执行，但 Gateway 起不来”或者“模型没响应”这类问题，第一反应就该是先跑一次它。

## G1.1.3 首次引导：运行 `openclaw onboard`

第一次使用时，不建议手写配置文件，直接走新手引导（Onboarding Wizard）最快：

```bash
openclaw onboard
```

这个向导通常会带你走完几件事：

1. 创建 OpenClaw 的本地工作目录
2. 选择一个默认模型或认证方式
3. 初始化基础配置
4. 询问是否马上连接通道

本章只做“最小可运行”配置，所以到了通道这一步可以先跳过，后面 G1.2 再单独连接 Telegram 和 Discord。

> **衍生解释 — 新手引导（Onboarding Wizard）**
>
> Wizard 不是某种“AI 魔法”，它本质上是一个交互式配置流程。和手工编辑配置文件相比，Wizard 的价值在于：它知道哪些字段是必填，知道哪些选项之间互相依赖，也知道哪些配置应该落到 `~/.openclaw/` 目录下。你可以把它理解成“带校验能力的安装脚本”。

## G1.1.4 启动 Gateway 并发出第一条消息

向导结束后，先启动 Gateway 网关：

```bash
openclaw gateway
```

如果你想看得更细一点，可以带上 `--verbose`：

```bash
openclaw gateway --verbose
```

保持这个终端窗口不要关，再开一个新终端，发送第一条消息：

```bash
openclaw agent --message "你好，介绍一下你自己"
```

有些旧截图或非正式示例会把它写成 `openclaw agent "..."`，但以当前 CLI 为准，实际要写 `--message`。

如果一切正常，你会在终端里看到模型返回的回复内容。也可以先确认 Gateway 确实在线：

```bash
openclaw gateway status
```

## G1.1.5 这条消息到底经历了什么

第一次成功后，建议你别急着进入下一节，先把这条链路在脑子里过一遍：

```typescript
const firstTurn = {
  step1: "Gateway 启动并监听本地请求",
  step2: "agent 命令把消息发给 Gateway",
  step3: "Gateway 为这次对话选择 Agent、模型和会话",
  step4: "模型生成回复",
  step5: "Gateway 把结果回传给 CLI 并显示在终端",
};
```

换句话说，你刚才看到的不是“CLI 自己会聊天”，而是：

- Gateway 网关先启动
- `agent` 命令把消息路由给 Gateway
- Gateway 再把请求交给 AI 模型
- 模型返回结果
- CLI 把结果显示出来

这个分层很重要。后面你接 Telegram、Discord、Cron、Webhook 的时候，表面上入口变了，但中间的大脑仍然是 Gateway。Gateway 的详细架构见第 3 章。

## G1.1.6 遇到问题先怎么查

最常见的几个排查动作，可以直接记住：

```bash
openclaw doctor
openclaw gateway status
openclaw models list
openclaw channels status
```

如果 `agent` 没回复，通常先看三件事：

- Gateway 有没有真的启动
- 模型认证或默认模型有没有配好
- 本地配置文件有没有被写坏

> **提示**
>
> 学习 OpenClaw 时，尽量把“安装成功”的标准定得具体一点：不是“命令装上了”，而是“`gateway` 能启动，`agent --message` 能收到回复”。后面很多问题，都是因为只完成了前一半。

## 本节小结

1. OpenClaw 的最低门槛不高：Node.js 22+ 加一台 macOS/Linux/WSL 机器就能开始。
2. 第一次安装最省事的路径是 `npm install -g openclaw` 然后运行 `openclaw onboard`。
3. 本章先跳过通道连接，先把 `openclaw gateway` 和 `openclaw agent --message "..."` 这条最小链路跑通。
4. 第一条消息的真实路径是“CLI -> Gateway -> 模型 -> Gateway -> CLI”，不是 CLI 直接调用模型。
5. 出现问题先跑 `openclaw doctor`，它是你排查环境问题的第一把锤子。
