# G3.2 高级自动化场景

> **生成模型**：Claude Opus 4.6 (anthropic/claude-opus-4-6)
> **Token 消耗**：输入 ~180k tokens，输出 ~7k tokens（本节）

---

## G3.2.1 从“能聊天”走向“能值班”

到了高级使用阶段，OpenClaw 的重点已经不是“把消息回复好”，而是“让代理主动工作”。最常见的入口有三类：**定时任务**（Cron）、**外部事件**（Webhook）、**邮箱推送**（Gmail Pub/Sub）。它们分别对应“按时间触发”“按系统事件触发”“按通信事件触发”。

> 衍生解释：如果把 Agent 看成函数，那么 Cron 决定何时调用，Webhook 决定谁来调用，Gmail Pub/Sub 决定哪些外部消息会变成调用参数。

## G3.2.2 场景一：自愈服务器

最典型的高级自动化，是做一个**自愈服务器**（Self-Healing Server）。思路很直接：每 5 分钟让 Agent 执行一句自然语言任务——“检查服务器状态，如果有异常就修复”。

CLI 方式可以直接这样建任务：

```bash
openclaw cron add \
  --name "Self-healing server" \
  --cron "*/5 * * * *" \
  --session isolated \
  --message "检查服务器状态，如果有异常就修复；检查磁盘、内存、进程、systemd 服务，并把处理结果发给我。" \
  --announce \
  --channel discord \
  --to "123456789012345678"
```

运行时，Agent 会通过 **Bash 工具**（Bash Tool）执行真实命令，例如：

```bash
df -h
free -m
ps aux
systemctl --failed
journalctl -u nginx -n 80 --no-pager
```

发现问题后，它可以继续执行修复命令，比如清理日志、重启服务、杀掉异常进程，然后把摘要投递给你。对应配置可以写成：

```json5
{
  cron: {
    enabled: true,
    webhookToken: "replace-with-dedicated-webhook-token",
    sessionRetention: "24h"
  },
  channels: {
    discord: {
      defaultAccount: "default"
    }
  }
}
```

如果你不想发到聊天频道，也可以把结果 POST 到你的告警系统：

```json5
{
  cron: {
    enabled: true,
    webhookToken: "replace-with-dedicated-webhook-token"
  }
}
```

这时把任务建成 `delivery.mode = "webhook"` 即可，结果会以 HTTP POST 的形式送到你的告警平台。

> 提示：这类任务最好使用 `--session isolated`，避免运维巡检把主对话历史弄得很脏。

## G3.2.3 场景二：自动化 DevOps 流水线

第二个高频场景，是把 GitHub Push 事件接进 OpenClaw，做成轻量 **自动化 DevOps 流水线**（Automated DevOps Pipeline）。例如：仓库发生 push 后，Webhook 把事件打到 OpenClaw；代理收到事件后运行测试、构建镜像、执行部署脚本。

先开启 Hook 入口：

```json5
{
  hooks: {
    enabled: true,
    token: "${OPENCLAW_HOOKS_TOKEN}",
    path: "/hooks",
    defaultSessionKey: "hook:ci",
    allowRequestSessionKey: false,
    allowedAgentIds: ["ops", "coder"]
  }
}
```

如果你希望不同事件自动映射到不同代理，还可以加一个显式映射：

```json5
{
  hooks: {
    enabled: true,
    token: "${OPENCLAW_HOOKS_TOKEN}",
    mappings: [
      {
        match: { path: "github-deploy" },
        action: "agent",
        wakeMode: "now",
        name: "GitHub",
        sessionKey: "hook:github:deploy",
        messageTemplate: "收到 GitHub push 事件，请拉取最新代码，运行测试、构建并部署，然后返回结果摘要。",
        deliver: true,
        channel: "last"
      }
    ]
  }
}
```

然后外部系统就可以调用：

```bash
curl -X POST http://127.0.0.1:18789/hooks/agent \
  -H 'Authorization: Bearer SECRET' \
  -H 'Content-Type: application/json' \
  -d '{"message":"GitHub push 事件已到达，请拉取最新代码，运行测试、构建并部署","agentId":"ops","name":"GitHub"}'
```

代理拿到事件后，通常会执行这类命令：

```bash
git pull --ff-only
pnpm install --frozen-lockfile
pnpm test
pnpm build
docker compose up -d --build
```

这种方案的好处是：CI/CD 不再局限于固定 YAML，你可以让 Agent 根据上下文决定“先回滚还是先重试”。

## G3.2.4 场景三：邮件自动处理

第三类场景是**邮件自动处理**。这里最适合用 Gmail Pub/Sub：新邮件到来后，Gmail 推送消息，OpenClaw 自动读取邮件摘要、判断类别、起草回复，必要时再把草稿发给你确认。

推荐先写入 Gmail Hook 配置：

```json5
{
  hooks: {
    enabled: true,
    token: "${OPENCLAW_HOOKS_TOKEN}",
    presets: ["gmail"],
    gmail: {
      account: "openclaw@gmail.com"
    }
  }
}
```

官方推荐流程是：

```bash
openclaw webhooks gmail setup \
  --account openclaw@gmail.com

openclaw webhooks gmail run
```

底层等价于把 `gog gmail watch serve` 接到 OpenClaw 的 `/hooks/gmail`：

```bash
gog gmail watch serve \
  --account openclaw@gmail.com \
  --bind 127.0.0.1 \
  --port 8788 \
  --path /gmail-pubsub \
  --token <shared> \
  --hook-url http://127.0.0.1:18789/hooks/gmail \
  --hook-token OPENCLAW_HOOK_TOKEN \
  --include-body
```

对于咨询类、客服类、招聘类邮箱，这个模式尤其有用：先分类，再摘要，再生成草稿，最后投递到 Slack 或 Discord 给你人工拍板。

## G3.2.5 场景四：定时报告生成

最后一个很实用的模式，是**定时报告生成**（Scheduled Report Generation）。比如每天早上 8 点，Agent 自动从若干 API 拉数据，汇总为日报，再发到 Slack 或 Discord。

```bash
openclaw cron add \
  --name "Daily metrics report" \
  --cron "0 8 * * *" \
  --session isolated \
  --message "抓取昨日销售、错误率、活跃用户和部署记录，整理成日报并发送。必要时用 Canvas 生成可视化卡片。" \
  --announce \
  --channel slack \
  --to "C0123456789"
```

如果数据结果适合视觉化展示，可以让 Agent 使用 **Canvas**（Canvas）生成图表、卡片式摘要或简单仪表板，再把结果发回协作频道。这样做比纯文本更适合团队同步。

## G3.2.6 多机协同时怎么做

当你的自动化跨越多台机器时，可以把 OpenClaw 看成一个统一编排层：某些任务在主机上决策，某些任务在远端节点执行。如果你后续接入 **OpenCrew** 一类的多机协同层，也可以把这些 Cron、Webhook、邮件任务继续上推到跨机器编排；但对大多数高级用户来说，先把单 Gateway 的自动化链路跑稳更重要。

## 本节小结

高级自动化的核心，不是把命令写得更长，而是把触发源接进来：Cron 适合自愈巡检和日报生成，Webhook 适合 GitHub 驱动的流水线，Gmail Pub/Sub 适合邮件分类与自动草拟。配合 Bash、Canvas 和多 Agent 路由，OpenClaw 可以从聊天助手升级为真正的自动化执行层。Cron 系统详见第 18 章，Webhook 钩子详见第 22 章。
