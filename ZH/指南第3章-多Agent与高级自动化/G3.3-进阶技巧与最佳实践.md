# G3.3 进阶技巧与最佳实践

> **生成模型**：Claude Opus 4.6 (anthropic/claude-opus-4-6)
> **Token 消耗**：输入 ~180k tokens，输出 ~7k tokens（本节）

---

## G3.3.1 先把“能跑”升级成“跑得久”

高级用户最容易踩的坑，不是功能不会配，而是系统跑久了之后开始变慢、丢上下文、权限过大或者部署不稳。所以这一节我们不讲炫技，而是讲一套长期可用的操作习惯。

## G3.3.2 记忆优化：让代理真的记住事

OpenClaw 的记忆系统本质上是 Markdown 文件。长期记忆放在 `MEMORY.md`，日常记录放在 `memory/YYYY-MM-DD.md`。如果你明确告诉代理“记住这件事”，它应该把内容写进磁盘，而不是只留在当前上下文里。

例如：

```bash
openclaw agent --message "记住：我的服务器IP是 1.2.3.4"
```

当你后面再问部署问题时，代理会先通过 **记忆搜索**（Memory Search）检索相关内容。为了提升召回效果，建议开启向量记忆：

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai"
      }
    }
  }
}
```

如果你已经配置了本地模型，也可以改成本地或其他嵌入提供者。重点不是一定用哪家，而是让“长期事实”可搜索、可复用。

> 衍生解释：向量记忆可以理解成“按语义查找”，不是靠关键词完全一致。你写的是“公网地址”，提问时说“服务器 IP”，系统也可能召回同一条记忆。

## G3.3.3 安全加固：默认假设输入不可信

高级自动化通常会连到群聊、Webhook、邮件和命令行，所以安全边界必须提早收紧。第一步是给非主会话启用**沙箱模式**（Sandbox Mode）：

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",
        workspaceAccess: "rw"
      }
    }
  }
}
```

`"non-main"` 的意思是：主会话仍可在宿主机运行，但群聊、Webhook、Cron 等非主会话默认进入沙箱。第二步是保留 **配对码**（Pairing Code）机制，不要为了省事把 DM 全开。第三步是给工具做允许/拒绝名单，限制危险命令：

```json5
{
  agents: {
    list: [
      {
        id: "ops",
        tools: {
          allow: ["bash", "read", "write"],
          deny: ["browser", "gateway", "nodes"]
        }
      }
    ]
  }
}
```

配完之后，跑一次审计：

```bash
openclaw security audit --deep
openclaw doctor
```

## G3.3.4 性能调优：贵模型留给关键路径

性能调优的第一原则是：不要让所有请求都打到最贵模型。比较稳妥的办法，是配置**模型回退链**（Model Fallback Chain）：

```json5
{
  agents: {
    defaults: {
      model: {
        primary: "anthropic/claude-opus-4-6",
        fallbacks: ["anthropic/claude-sonnet-4-5", "openai/gpt-5.2"]
      }
    }
  }
}
```

长对话里，另一个关键是**上下文裁剪**（Context Pruning）和**压缩**（Compaction）。当会话太长时，OpenClaw 会自动压缩旧上下文；你也可以显式配置：

```json5
{
  agents: {
    defaults: {
      compaction: {
        reserveTokensFloor: 20000,
        memoryFlush: {
          enabled: true,
          softThresholdTokens: 4000
        }
      }
    }
  }
}
```

必要时再手动触发：

```bash
/compact
```

如果你的聊天界面感觉“回复太碎、太拖”，可以启用 **分块流式**（Block Streaming）来改善交互体验：

```json5
{
  channels: {
    telegram: {
      streaming: "block"
    },
    discord: {
      streaming: "block"
    }
  }
}
```

对 Telegram、Discord 这类通道，块级输出通常比字符级流式更稳，也更像人在发消息。

## G3.3.5 生产部署：优先选 Docker

当你准备长期运行时，Docker 往往比手工守护进程更稳。核心思路只有三点：用 `docker-compose.yml` 管服务；把配置目录挂载出来；启用自动重启。

```yaml
services:
  openclaw:
    restart: unless-stopped
    volumes:
      - ~/.openclaw:/home/node/.openclaw
      - ~/openclaw-workspace:/workspace
```

这里的挂载很关键：如果不持久化，模型配置、会话、记忆、缓存都会在容器重建后丢失。常用命令如下：

```bash
docker compose up -d
docker compose logs -f
docker compose restart
```

## G3.3.6 定制你的 Agent：`AGENTS.md` 与 `SOUL.md`

很多高级用户忽略了工作区级定制。其实 `AGENTS.md` 更适合放项目规则，比如代码规范、目录约定、发布流程；`SOUL.md` 更适合放人格风格、说话语气、协作习惯。它们会在会话启动时被注入系统提示。

一个实用做法是：每个工作区都写自己的 `AGENTS.md`，把“这个仓库怎么改、哪些目录不能碰、测试命令是什么”写清楚；需要人格差异时，再给不同代理配不同的 `SOUL.md`。

## G3.3.7 常见坑与排查

最后给你一组高频排查路径：

- **Agent 没响应**：先跑 `openclaw doctor`，再检查模型 API Key、Gateway 是否在线。
- **消息太慢**：检查网络、模型供应商状态，并优先启用块级流式输出。
- **上下文丢失**：看是不是会话被 reset，或根本没有把长期事实写入 `MEMORY.md`。
- **Webhook/Cron 没触发**：检查 `hooks.enabled`、`cron.enabled`、令牌、时区和反向代理。

很多时候，问题不在模型，而在“上下文是否持久化、权限是否合理、触发器是否真的连通”。

## 本节小结

真正稳定的 OpenClaw，靠的是三层功夫：记忆要可检索，安全边界要默认收紧，性能与部署要按长期运行来设计。把 `MEMORY.md`、向量记忆、`sandbox.mode: "non-main"`、模型回退链、Docker 持久化和 `AGENTS.md`/`SOUL.md` 这几件事做好，你的自动化系统就会从“偶尔能用”升级到“可以托付”。安全模型详见第 24 章，配置系统详见第 23 章，部署方案详见第 29 章。
