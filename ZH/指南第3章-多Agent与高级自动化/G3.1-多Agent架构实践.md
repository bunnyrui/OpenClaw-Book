# G3.1 多 Agent 架构实践

> **生成模型**：Claude Opus 4.6 (anthropic/claude-opus-4-6)
> **Token 消耗**：输入 ~180k tokens，输出 ~7k tokens（本节）

---

## G3.1.1 为什么要拆成多个 Agent

当你开始用 OpenClaw 做复杂自动化时，单个 Agent 往往不够用了。原因不是模型“不聪明”，而是任务边界不同：写代码、查资料、巡检服务器，本来就是三种工作流。把它们塞进同一个会话里，容易出现上下文污染、工具权限过大、人格指令打架等问题。

更实用的做法是做成多个**代理**（Agent）：让 **编码代理**（Coding Agent）负责改代码和跑测试，让 **研究代理**（Research Agent）负责查文档和网页，让 **运维代理**（Ops Agent）负责部署、监控与修复。这样做有三个直接收益：一是上下文更干净；二是权限可以分层；三是模型可以按任务选型，不必所有事都上最贵模型。

> 提示：这里的“多 Agent”不是花哨概念，而是把“不同岗位的人”映射成不同工作目录、不同技能集、不同会话空间。

## G3.1.2 基础配置：`agents.list[]`

OpenClaw 用 `agents.list[]` 定义多个代理。每个代理都可以有自己的模型、技能、工作区和状态目录。下面这个配置很适合高级用户起步：

```json5
{
  agents: {
    list: [
      {
        id: "coder",
        name: "Code Assistant",
        model: "anthropic/claude-sonnet-4-20250514",
        skills: ["coding-agent", "github"],
        workspace: "~/projects"
      },
      {
        id: "researcher",
        name: "Research Bot",
        model: "google/gemini-2.5-pro",
        skills: ["browser"]
      },
      {
        id: "ops",
        name: "DevOps Agent",
        model: "openai/gpt-4o",
        skills: ["server-monitor"]
      }
    ]
  }
}
```

你也可以先用 CLI 建代理，再回头微调配置：

```bash
openclaw agents add coder
openclaw agents add researcher
openclaw agents add ops
openclaw agents list --bindings
```

## G3.1.3 路由：把消息送给正确的代理

多 Agent 真正的关键，不是“配出来”，而是“路由对”。OpenClaw 用 `bindings` 做确定性路由，常见有三种。

第一种是**按频道**（Per-channel）路由，例如“Discord 的消息都交给 coder”：

```json5
{
  bindings: [
    { agentId: "coder", match: { channel: "discord", accountId: "*" } }
  ]
}
```

第二种是**按群组或对端**（Per-group / Per-peer）路由。比如某个 Telegram 群专门让 researcher 负责：

```json5
{
  bindings: [
    {
      agentId: "researcher",
      match: { channel: "telegram", peer: { kind: "group", id: "-1001234567890" } }
    }
  ]
}
```

第三种是多 Agent 频道里的 **@提及切换**。实战里很常见：默认由 coder 回答；当你在群里 `@researcher` 时，把当前问题交给 researcher；需要部署时再 `@ops`。这种方式适合共享频道，因为你不用切频道，就能显式指定“这条消息要给谁处理”。

> 衍生解释：这里的“路由”可以类比 Web 框架的 URL 分发。消息先匹配 `peer`，再匹配 `accountId`、`channel`，最后才回落到默认代理。

## G3.1.4 子代理：主代理把子任务外包出去

多 Agent 不只体现在入口路由。更强的一点，是主代理可以在执行过程中拉起**子代理**（Sub-agent）。OpenClaw 提供 `sessions_spawn` 工具，让一个正在运行的代理把子任务委托出去。

典型流程如下：

```typescript
// coder 收到需求后
// 1. 自己负责拆解与实现
// 2. 用 sessions_spawn 拉起 researcher
// 3. researcher 查官方文档/接口差异
// 4. coder 拿到结果后继续改代码并总结
```

主代理与子代理之间不会共享同一个会话上下文，它们是隔离的。这一点非常重要：研究型任务会产生很多网页碎片，如果直接塞进编码上下文，代码质量通常会下降。

## G3.1.5 跨 Agent 协作：会话级通信

如果你希望多个代理之间显式交换信息，OpenClaw 还提供了三类会话通信工具：

- `sessions_list`：列出当前可见会话
- `sessions_history`：读取其他会话的历史摘要或消息
- `sessions_send`：向指定会话发送消息

这让你可以做出更像“团队协作”的流程。例如 researcher 先独立跑一轮资料收集，然后 coder 通过 `sessions_history` 看结果，再决定是否采纳；或者 ops 在部署完后用 `sessions_send` 把回滚建议发回 coder 会话。

## G3.1.6 一个完整场景：编码代理 + 研究子代理

假设你在 Discord 里对 coder 说：“给项目加一个 GitHub Webhook 自动部署功能。”合理流程通常是这样的：

1. coder 判断这是编码任务，先在本地工作区分析仓库；
2. coder 发现 GitHub Webhook 细节不确定，于是调用 `sessions_spawn` 启动 researcher；
3. researcher 去查 GitHub 事件格式、OpenClaw 的 Hook 配置、已有部署脚本；
4. researcher 把发现结果返回；
5. coder 根据结果修改代码、补配置、运行测试；
6. 如需上线，再把部署动作交给 ops。

你也可以从 CLI 直接给某个代理发起任务：

```bash
openclaw agent --message "为当前仓库增加 webhook 自动部署能力" --thinking high
openclaw gateway restart
```

这里的重点不是“用了几个模型”，而是**谁负责决策，谁负责查证，谁负责落地**。一旦这三层分清，多 Agent 就会非常稳定。

## 本节小结

多 Agent 的价值，在于把复杂自动化拆成多个职责清晰的工作单元。你可以先用 `agents.list[]` 定义编码、研究、运维三个代理，再通过 `bindings` 把不同频道、群组和联系人绑定到合适的代理；需要更细粒度协作时，再用 `sessions_spawn`、`sessions_list`、`sessions_history`、`sessions_send` 组合出真正的代理协同流程。多代理架构的源码分析见第 31 章。
