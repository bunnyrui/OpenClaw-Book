Starting from this chapter, we formally enter the source code analysis of OpenClaw's most critical subsystem—the Gateway Control Plane. Gateway is the heart of the entire OpenClaw system, and understanding its design philosophy and implementation details is key to understanding the entire project.

## 3.1.1 The Meaning of "Control Plane"

> **Sidebar**: Control Plane is a concept originating from network engineering and distributed systems. In computer networks, the control plane is responsible for decision-making—determining where data packets should be forwarded; while the data plane is responsible for execution—actually forwarding the data packets. Similarly, in microservice architectures, the control plane (such as Kubernetes API Server) is responsible for management and coordination, while individual worker nodes are responsible for actual task execution.
>
> In OpenClaw, Gateway plays the role of the control plane. It does not itself perform AI inference (that is the work of the Pi Agent runtime), nor does it directly handle message format conversion (that is the work of channel adapters). What it does is **coordination**: receiving requests and events from all parties, making routing decisions, scheduling execution, and distributing results.

Gateway's core responsibilities can be divided into six major domains:

### 1. Connection Management

Gateway maintains two fundamentally different types of connections, and understanding their distinction is crucial:

**Channels — Embedded Message Adapters**

Channels are **not** external services that connect to Gateway over the network—they are plugins/adapters **embedded within the Gateway process**. When Gateway starts, it calls `startChannels()` (`src/gateway/server-channels.ts`) to initialize all configured channels. Communication between channels and Gateway is via **in-process function calls**.

Each channel is responsible for communicating with its corresponding external messaging platform:
- **Telegram**: Uses the grammY library to communicate with Telegram servers via HTTP long polling (or Webhook)
- **WhatsApp**: Uses the Baileys library to communicate with WhatsApp servers via WebSocket
- **Discord**: Uses the discord.js library to communicate with Discord gateway via WebSocket
- **Slack**: Uses the Bolt library to communicate with Slack via WebSocket (Socket Mode) or HTTP events

Note: The "communication" here refers to network communication between channels and **external platform servers**. There is **no** network communication between channels and Gateway—they are in the same Node.js process, passing messages through function calls.

**Clients — WebSocket Remote Connections**

Clients are **external programs** that connect to Gateway via the WebSocket protocol, listening on port `18789`. Clients are divided into two roles:

- **Operators**: macOS menu bar app, CLI tools, Web console. Used for monitoring Agent execution status, approving command execution, managing sessions and configuration.
- **Nodes**: macOS/iOS/Android device nodes. Used to expose device capabilities to Agents—camera, screen recording, location, Canvas surface, etc.

Each client connection has authentication, permission levels, and event subscription status. Gateway tracks all active connections and pushes events to relevant parties when status changes.

> **Key Distinction**: In simple message Q&A scenarios (e.g., user sends a message on Telegram, Agent replies), clients are completely uninvolved. The message enters Gateway from the channel → is routed to the Agent → Agent generates a reply → reply is sent back to the user through the channel—the entire process completes within the Gateway process. Clients only participate in these scenarios:
> - Agent needs to execute a dangerous command, requiring operator approval (exec approval)
> - Agent needs to invoke device capabilities (e.g., taking photos, screen recording), requiring node cooperation
> - Operator actively viewing Agent execution status or managing sessions

### 2. Message Routing

When a message arrives from any channel, Gateway needs to decide:

- Which Agent should process this message? (multi-Agent routing)
- Which session should be used? (session key generation)
- Is there currently a running Agent in that session? (queue management)
- Which channel should the reply be sent back to? (outbound routing)

### 3. Session Management

A session is a container for conversation context in OpenClaw. Gateway manages the complete lifecycle of sessions: creation, read/write, reset, expiration, compaction, deletion.

### 4. Agent Scheduling

When AI processing is needed, Gateway launches Agent runs, manages queues, timeouts, and cancellations. The results of Agent runs are pushed to all interested clients through the event stream.

### 5. Tool Infrastructure

Certain tools (such as Cron scheduling, browser control, node invocation) require support at the Gateway layer. Gateway provides the infrastructure needed by these tools.

### 6. Configuration and Operations

Gateway is responsible for loading, validating, and hot-reloading configuration, providing health check endpoints, and managing daemon process lifecycle.

## 3.1.2 Single Gateway Instance Constraint

An important design constraint of OpenClaw is: **only one Gateway instance runs per host**. This is explicitly defined as an invariant in `docs/concepts/architecture.md`:

> *"Exactly one Gateway controls a single Baileys session per host."*

This constraint stems from WhatsApp's limitations—Baileys (an unofficial implementation of the WhatsApp Web protocol) only allows one active Web session. If two processes simultaneously open a Web session for the same WhatsApp account, the previous one will be kicked offline.

OpenClaw enforces this constraint through file locks and port exclusivity. If you try to start a second Gateway on the same machine, it will detect that the port is already in use and report an error. The relevant implementation is in `src/infra/ports.ts`:


## 3.1.3 Gateway's Responsibility Boundaries

Understanding what Gateway **does not do** is equally important:

| Gateway Does | Gateway Does Not Do |
|-------------|---------------|
| WebSocket/HTTP server management | AI model inference (delegated to Pi Agent) |
| Message routing and session key parsing | Channel protocol implementation (delegated to channel adapters) |
| Agent run scheduling and queue management | Specific tool execution (delegated to tool system) |
| Event broadcasting and client notification | Native application UI rendering (delegated to clients) |
| Configuration loading and hot-reloading | Model selection and Auth Profile management (delegated to Agent) |
| Security authentication and device pairing | Markdown rendering (delegated to channel outbound processing) |

This clear responsibility boundary means that although Gateway has many files (128 files), each file has a relatively single responsibility. Taking the file naming in `src/gateway/` as an example:

```
server-channels.ts      → Manage channel connections
server-chat.ts          → Handle chat messages/Agent events
server-cron.ts          → Manage Cron scheduling
server-http.ts          → HTTP routing
server-ws-runtime.ts    → WebSocket runtime
server-lanes.ts         → Queue lanes
server-broadcast.ts     → Event broadcasting
server-discovery.ts     → Service discovery
server-tailscale.ts     → Tailscale integration
server-maintenance.ts   → Periodic maintenance tasks
server-startup.ts       → Startup sequence
server-close.ts         → Shutdown sequence
```

Each filename intuitively reflects its responsibility. This naming convention is consistent throughout the entire OpenClaw codebase.
