Discord uses **Snowflake IDs** as unique identifiers for all entities (users, messages, channels, servers)—a 64-bit integer encoding timestamp, worker node ID, and sequence number.

> **Sidebar: Snowflake ID**
>
> Snowflake is a distributed unique ID generation algorithm invented by Twitter (now X). A Snowflake ID contains: 41-bit timestamp (~69 years) + 10-bit machine ID + 12-bit sequence number. Discord adopted a similar scheme with its epoch starting January 1, 2015. This means Snowflake IDs are not only unique but also implicitly contain creation time information.

DM (private message) policy controls who can send private messages to the Bot:

| Policy | Description |
|--------|-------------|

Discord's mention format is `<@user_id>` or `<@!user_id>`, different from Telegram's `@username` and WhatsApp's `@phone_number`.

## 12.3.3 Native Slash Commands and Text Commands

Discord supports **native Slash Commands** (Application Commands), where users see a command list and parameter hints when typing `/` in the chat box. OpenClaw registers these commands via `createDiscordNativeCommand`.

Similar to Telegram's custom commands, Discord's Slash commands must also be registered with the Discord API, but the implementation is more complex—Discord requires creating commands through the Application Commands API and supports advanced features like subcommands, option types, and autocomplete.

OpenClaw's Discord channel also supports regular text commands (starting with a configured prefix), providing a compatible entry point for users unfamiliar with Slash commands.

## 12.3.4 Lines Per Message Limit

Discord clients have special rendering behavior for long messages—messages exceeding a certain line count are collapsed. OpenClaw's Discord chunker (`src/discord/chunk.ts`) considers both **character count** and **line count** dimensions:

```typescript
// src/discord/chunk.ts (simplified)
const DEFAULT_MAX_CHARS = 2000;    // Discord message character limit
const DEFAULT_MAX_LINES = 17;      // Soft line count limit

export function chunkDiscordText(text, opts): string[] {
  const maxChars = opts.maxChars ?? DEFAULT_MAX_CHARS;
  const maxLines = opts.maxLines ?? DEFAULT_MAX_LINES;

  // Fast path
  if (body.length <= maxChars && countLines(body) <= maxLines) {
    return [body];
  }

  // Process line by line, tracking code block (fence) state
  let openFence: OpenFence | null = null;

  for (const line of lines) {
    // Detect code block open/close
    const fenceInfo = parseFenceLine(line);
    if (fenceInfo) {
      if (!openFence) openFence = fenceInfo;         // Open code block
      else if (fenceInfo.markerLen >= openFence.markerLen)
        openFence = null;                             // Close code block
    }

    // Chunk when exceeding character/line limits
    if ((wouldExceedChars || wouldExceedLines) && current.length > 0) {
      flush();    // Automatically close/reopen code blocks on flush
    }
    // ...
  }
}
```

The most elegant design of this chunker is **code block balancing**: when the chunk point falls inside a code block, it inserts a closing marker (` ``` `) at the end of the current chunk and reopens the code block at the start of the next chunk:

```
Original message:
  ```python
  def hello():
      print("hello")
  def world():      ← Chunk point here
      print("world")
  ```

After chunking:
  Chunk 1: ```python
           def hello():
               print("hello")
           ```            ← Auto-closed

  Chunk 2: ```python      ← Auto-reopened
           def world():
               print("world")
           ```
```

There's also a `rebalanceReasoningItalics` function that specifically handles italic marker balancing for reasoning text—when reasoning text (italic wrapped in `_`) is chunked, it ensures each chunk renders correctly independently.
