## 28.1.3 Voice Wake + Push-to-Talk

### Voice Wake System

Voice wake is one of the core features of the macOS application. It uses Apple's **Speech framework** (`SFSpeechRecognizer`) for real-time speech recognition, listening for user-defined trigger words.

```swift
// apps/macos/Sources/OpenClaw/VoiceWakeRuntime.swift
actor VoiceWakeRuntime {
    static let shared = VoiceWakeRuntime()
    
    enum ListeningState { case idle, voiceWake, pushToTalk }
    
    private var recognizer: SFSpeechRecognizer?
    private var audioEngine: AVAudioEngine?
    private var recognitionRequest: SFSpeechAudioBufferRecognitionRequest?
    // ...
    
    // Tunable parameters
    private let silenceWindow: TimeInterval = 2.0        // Silence threshold after speech
    private let triggerOnlySilenceWindow: TimeInterval = 5.0
    private let captureHardStop: TimeInterval = 120.0     // Maximum capture duration
    private let minSpeechRMS: Double = 1e-3               // Minimum speech RMS
    private let speechBoostFactor: Double = 6.0           // Multiplier above noise floor
}
```

> **Sidebar — SFSpeechRecognizer and RMS**
>
> `SFSpeechRecognizer` is Apple's speech recognition API that supports real-time streaming recognition. It converts audio buffers captured from the microphone (`AVAudioPCMBuffer`) into text. RMS (Root Mean Square) is a common metric for measuring audio signal strength—OpenClaw determines whether the user is speaking by comparing the current RMS to the noise floor (`noiseFloorRMS`).

The voice wake flow:

1. Start `AVAudioEngine` to continuously capture audio
2. Feed audio buffers into `SFSpeechAudioBufferRecognitionRequest` for real-time recognition
3. In the recognition callback, check if the text contains trigger words (e.g., "Hey OpenClaw")
4. After triggering, enter "capture mode"—continue recording until 2 seconds of silence is detected
5. Send the captured speech to the Gateway

The audio engine is **lazily created**: `AVAudioEngine` is only initialized when the voice wake feature is actually enabled, because merely creating it causes Bluetooth headsets to switch to a lower-quality headset profile.

## 28.1.6 macOS Permissions and TCC

macOS has a strict **TCC (Transparency, Consent, and Control)** permission system. OpenClaw requires multiple system permissions to work fully:

```swift
// apps/macos/Sources/OpenClaw/PermissionManager.swift
enum PermissionManager {
    static func ensure(_ caps: [Capability], interactive: Bool) async -> [Capability: Bool] {
        var results: [Capability: Bool] = [:]
        for cap in caps {
            results[cap] = await ensureCapability(cap, interactive: interactive)
        }
        return results
    }
}
```

Required permissions include:

| Permission | Purpose | API |
|------------|---------|-----|
| Microphone | Voice wake/conversation mode | `AVCaptureDevice.requestAccess(for: .audio)` |
| Speech Recognition | Speech-to-text | `SFSpeechRecognizer.requestAuthorization()` |
| Camera | Vision capabilities | `AVCaptureDevice.requestAccess(for: .video)` |
| Screen Recording | Screenshot/screen recording tools | `CGPreflightScreenCaptureAccess()` |
| AppleScript | Automation operations | AppleScript permission |
| Accessibility | Keyboard listening | Accessibility API |
| Notifications | Message alerts | `UNUserNotificationCenter.requestAuthorization()` |
| Location | Location query tools | `CLLocationManager.requestAlwaysAuthorization()` |

> **Sidebar — TCC (Transparency, Consent, and Control)**
>
> TCC is the privacy protection framework for macOS and iOS. Applications must obtain explicit user authorization before accessing sensitive resources (camera, microphone, folders, etc.). Permission states are stored in the system database `TCC.db`, and users can manage them in the "Privacy & Security" panel of System Settings. For developers, you must declare usage descriptions in Info.plist (e.g., `NSMicrophoneUsageDescription`), otherwise permission requests will fail silently.

`PermissionManager` in interactive mode guides users through authorization (showing system dialogs or navigating to Settings), while in non-interactive mode it only checks without requesting.
