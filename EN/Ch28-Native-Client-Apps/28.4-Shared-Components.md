There is substantial shared logic between the macOS and iOS applications—Gateway WebSocket protocol, chat UI, Canvas command parsing, device authentication, and more. This shared code is unified in `apps/shared/OpenClawKit/`.

## 28.4.1 OpenClawKit Architecture

### Swift Package Structure

OpenClawKit is a Swift package containing three library targets:

```swift
// apps/shared/OpenClawKit/Package.swift
let package = Package(
    name: "OpenClawKit",
    platforms: [.iOS(.v18), .macOS(.v15)],
    products: [
        .library(name: "OpenClawProtocol", targets: ["OpenClawProtocol"]),
        .library(name: "OpenClawKit", targets: ["OpenClawKit"]),
        .library(name: "OpenClawChatUI", targets: ["OpenClawChatUI"]),
    ],
    dependencies: [
        .package(url: "ElevenLabsKit", exact: "0.1.0"),   // TTS speech synthesis
        .package(url: "textual", exact: "0.3.1"),          // Markdown rendering

)
```

#### WebSocket Client

`GatewayChannel` is the shared WebSocket client, decoupled from concrete network implementations through protocol abstraction (`WebSocketTasking`):

```swift
// apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayChannel.swift
public protocol WebSocketTasking: AnyObject {
    var state: URLSessionTask.State { get }
    func resume()
    func cancel(with closeCode: URLSessionWebSocketTask.CloseCode, reason: Data?)
    func send(_ message: URLSessionWebSocketTask.Message) async throws
    func receive() async throws -> URLSessionWebSocketTask.Message
}

extension URLSessionWebSocketTask: WebSocketTasking {}
```

The default `URLSession` implementation sets a maximum message size of 16MB—large snapshots and history payloads may exceed the default limit:

```swift
extension URLSession: WebSocketSessioning {
    public func makeWebSocketTask(url: URL) -> WebSocketTaskBox {
        let task = self.webSocketTask(with: url)
        task.maximumMessageSize = 16 * 1024 * 1024  // 16 MB
        return WebSocketTaskBox(task: task)
    }
}
```

Connection options are passed through the `GatewayConnectOptions` struct:

```swift
public struct GatewayConnectOptions: Sendable {
    public var role: String          // "node" or "operator"
    public var scopes: [String]      // Permission scopes
    public var caps: [String]        // Device capabilities
    public var commands: [String]    // Supported commands
    public var permissions: [String: Bool]  // Acquired permissions
    public var clientId: String
    public var clientMode: String
}
```

#### Capability System

Device capabilities are defined through a concise enum:

```swift
public enum OpenClawCapability: String, Codable, Sendable {
    case canvas       // Canvas manipulation
    case camera       // Camera
    case screen       // Screen recording
    case voiceWake    // Voice wake
    case location     // Location information
}
```

Each capability corresponds to a set of command types (such as `CameraCommands`, `ScreenCommands`). The Gateway decides which tools to expose to AI agents based on the list of capabilities declared by the node.

### Android's Sharing Strategy

The Android application cannot directly use Swift packages. Its sharing strategy is:

1. **Resource file reuse**: Directly reference `OpenClawKit/Resources` through Gradle's `assets.srcDir`
2. **Protocol reimplementation**: Capability enums, command types, etc. are redefined in Kotlin in the `ai.openclaw.android.protocol` package
3. **Communication protocol consistency**: Although the languages differ, the JSON format of WebSocket frames is unified across platforms

## Summary of This Section

1. **OpenClawKit is a shared Swift package for macOS and iOS**, adopting a three-layer architecture: Protocol (pure data) → Kit (core logic) → ChatUI (SwiftUI components).
2. **WebSocket client** is decoupled from URLSession through protocol abstraction, supporting 16MB large messages and TLS certificate pinning.
3. **Capability system** declares device hardware capabilities through 5 enum values, with the Gateway deciding available tools based on this.
4. **ChatUI layer** provides a complete SwiftUI chat interface, supporting different network backends through the `ChatTransport` protocol.
5. **Android maintains cross-platform consistency through resource sharing and protocol reimplementation**, with JSON frame format being the unified contract across all three platforms.
