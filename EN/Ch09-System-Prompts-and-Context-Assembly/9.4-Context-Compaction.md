# 9.4 Context Compaction

> **Generated by**: Claude Opus 4.6 (anthropic/claude-opus-4-6)
> **Token usage**: Input ~450,000 tokens, Output ~36,000 tokens (Chapter 9 total)

---

Large language models have an inherent limitation — the **context window**. Whether a model's context window is 8K, 128K, or 200K tokens, prolonged conversations will eventually exhaust available space. When the token count of conversation history approaches the context window limit, without intervention, the model API returns a "context overflow" error and the conversation is forced to terminate.

OpenClaw's **context compaction** mechanism elegantly solves this problem: before context overflow occurs, it **summarizes** old conversation history — replacing large amounts of original messages with a brief summary, thereby freeing up context space and allowing conversations to continue indefinitely.

> **Sidebar — Context Window**
>
> A context window is the maximum amount of text a large language model can "see" in a single inference, measured in tokens. One token roughly corresponds to 4 English characters or 1-2 Chinese characters. For example, Claude's context window is 200K tokens, meaning all messages in a conversation (including system prompt) cannot exceed approximately 200,000 tokens. Exceeding this limit, the API rejects the request.

## 9.4.1 Automatic Compaction Trigger Mechanism

### Trigger Timing: Context Overflow Detection

OpenClaw doesn't preemptively compact periodically — it uses a **passive trigger** strategy: compaction starts only when the model API returns a context overflow error.

In the main loop of `runEmbeddedPiAgent` (see Section 7.2), after each API call, the error type is checked:

```typescript
// src/agents/pi-embedded-runner/run.ts (simplified)
const MAX_OVERFLOW_COMPACTION_ATTEMPTS = 3;
let overflowCompactionAttempts = 0;

while (true) {
  const attempt = await runAttempt({ ... });
  
  if (promptError && !aborted) {
    const errorText = describeUnknownError(promptError);
    
    if (isContextOverflowError(errorText)) {
      const isCompactionFailure = isCompactionFailureError(errorText);
      
      // Only compact for "real" context overflow, not for compaction failure
      if (!isCompactionFailure && 
          overflowCompactionAttempts < MAX_OVERFLOW_COMPACTION_ATTEMPTS) {
        overflowCompactionAttempts++;
        log.warn(
          `context overflow detected (attempt ${overflowCompactionAttempts}/` +
          `${MAX_OVERFLOW_COMPACTION_ATTEMPTS}); attempting auto-compaction`
        );
        
        const compactResult = await compactEmbeddedPiSessionDirect({
          sessionId: params.sessionId,
          sessionFile: params.sessionFile,
          workspaceDir: resolvedWorkspace,
          provider,
          model: modelId,
          // ...other parameters
        });
        
        if (compactResult.compacted) {
          log.info("auto-compaction succeeded; retrying prompt");
          continue; // Compaction succeeded, retry current request
        }
        
        log.warn(`auto-compaction failed: ${compactResult.reason}`);
      }
      
      // All compaction attempts failed or not compactable
      return {
        payloads: [{
          text: "Context overflow: prompt too large for the model.",
          isError: true,
        }],
        meta: { error: { kind: isCompactionFailure 
          ? "compaction_failure" : "context_overflow" } },
      };
    }
  }
  // ...normal processing
}
```

Key design decisions:

1. **Maximum 3 compaction retries**: `MAX_OVERFLOW_COMPACTION_ATTEMPTS = 3`, preventing infinite loops when compaction is insufficient.
2. **Distinguish overflow from compaction failure**: If the error itself is "overflow caused by compaction failure" (`compaction_failure`), compaction won't be retried again, avoiding recursion issues.
3. **Retry after success**: After successful compaction, the current request is retried directly via `continue`, transparent to users.

## 9.4.2 Compaction Flow (`src/agents/compaction.ts`)

### Core Algorithm: Token Estimation and Message Chunking

The first step of compaction is estimating the token count of messages:

```typescript
// src/agents/compaction.ts
export function estimateMessagesTokens(messages: AgentMessage[]): number {
  return messages.reduce(
    (sum, message) => sum + estimateTokens(message), 0
  );
}
```

### Splitting Messages by Token Share

`splitMessagesByTokenShare` divides messages into multiple chunks by token count, with roughly equal token counts per chunk:

```typescript
export function splitMessagesByTokenShare(
  messages: AgentMessage[],
  parts = 2,  // Default split into 2 chunks
): AgentMessage[][] {
  if (messages.length === 0) return [];
  
  const normalizedParts = normalizeParts(parts, messages.length);
  if (normalizedParts <= 1) return [messages];
  
  const totalTokens = estimateMessagesTokens(messages);
  const targetTokens = totalTokens / normalizedParts;
  const chunks: AgentMessage[][] = [];
  let current: AgentMessage[] = [];
  let currentTokens = 0;
  
  for (const message of messages) {
    const messageTokens = estimateTokens(message);
    // Current chunk reached target size and more chunks remain → start new chunk
    if (chunks.length < normalizedParts - 1 &&
        current.length > 0 &&
        currentTokens + messageTokens > targetTokens) {
      chunks.push(current);
      current = [];
      currentTokens = 0;
    }
    current.push(message);
    currentTokens += messageTokens;
  }
  
  if (current.length > 0) chunks.push(current);
  return chunks;
}
```

### Chunking by Maximum Token Count

Another chunking approach is setting a maximum token limit per chunk:

```typescript
export function chunkMessagesByMaxTokens(
  messages: AgentMessage[],
  maxTokens: number,
): AgentMessage[][] {
  const chunks: AgentMessage[][] = [];
  let currentChunk: AgentMessage[] = [];
  let currentTokens = 0;
  
  for (const message of messages) {
    const messageTokens = estimateTokens(message);
    if (currentChunk.length > 0 && 
        currentTokens + messageTokens > maxTokens) {
      chunks.push(currentChunk);
      currentChunk = [];
      currentTokens = 0;
    }
    currentChunk.push(message);
    currentTokens += messageTokens;
    
    // Handle oversized messages: separate chunk
    if (messageTokens > maxTokens) {
      chunks.push(currentChunk);
      currentChunk = [];
      currentTokens = 0;
    }
  }
  
  if (currentChunk.length > 0) chunks.push(currentChunk);
  return chunks;
}
```

### Adaptive Chunking Ratio

When messages have large average size, a smaller chunking ratio is needed to avoid individual chunks exceeding model limits:

```typescript
export const BASE_CHUNK_RATIO = 0.4;    // Base: 40% of context window
export const MIN_CHUNK_RATIO = 0.15;     // Minimum: 15%
export const SAFETY_MARGIN = 1.2;        // 20% safety margin

export function computeAdaptiveChunkRatio(
  messages: AgentMessage[], 
  contextWindow: number
): number {
  if (messages.length === 0) return BASE_CHUNK_RATIO;
  
  const totalTokens = estimateMessagesTokens(messages);
  const avgTokens = totalTokens / messages.length;
  
  // Add safety margin (token estimation may be low)
  const safeAvgTokens = avgTokens * SAFETY_MARGIN;
  const avgRatio = safeAvgTokens / contextWindow;
  
  // If average message > 10% of context, reduce chunk ratio
  if (avgRatio > 0.1) {
    const reduction = Math.min(
      avgRatio * 2, 
      BASE_CHUNK_RATIO - MIN_CHUNK_RATIO
    );
    return Math.max(MIN_CHUNK_RATIO, BASE_CHUNK_RATIO - reduction);
  }
  
  return BASE_CHUNK_RATIO;
}
```

Example:

- **Normal conversation** (avg message ~500 tokens, 200K context): `avgRatio = 0.003`, far below 10%, use base ratio 40%
- **With large code blocks** (avg message ~30K tokens, 200K context): `avgRatio = 0.18`, use reduced ratio `max(0.15, 0.4 - 0.36) = 0.15`

### Oversized Message Detection

If a single message exceeds 50% of the context window, it cannot be safely summarized:

```typescript
export function isOversizedForSummary(
  msg: AgentMessage, 
  contextWindow: number
): boolean {
  const tokens = estimateTokens(msg) * SAFETY_MARGIN;
  return tokens > contextWindow * 0.5;
}
```

### Staged Summarization (summarizeInStages)

This is the core compaction function. It first chunks messages, generates summaries for each chunk separately, then merges all summaries into a comprehensive summary:

```typescript
export async function summarizeInStages(params: {
  messages: AgentMessage[];
  model: NonNullable<ExtensionContext["model"]>;
  apiKey: string;
  signal: AbortSignal;
  reserveTokens: number;
  maxChunkTokens: number;
  contextWindow: number;
  customInstructions?: string;
  previousSummary?: string;
  parts?: number;             // Number of chunks, default 2
  minMessagesForSplit?: number; // Min messages to split, default 4
}): Promise<string> {
  const { messages } = params;
  if (messages.length === 0) {
    return params.previousSummary ?? "No prior history.";
  }
  
  // Too few messages or tokens within limit → summarize directly
  const parts = normalizeParts(params.parts ?? 2, messages.length);
  const totalTokens = estimateMessagesTokens(messages);
  if (parts <= 1 || 
      messages.length < minMessagesForSplit || 
      totalTokens <= params.maxChunkTokens) {
    return summarizeWithFallback(params);
  }
  
  // Chunked summarization
  const splits = splitMessagesByTokenShare(messages, parts)
    .filter((chunk) => chunk.length > 0);
  if (splits.length <= 1) {
    return summarizeWithFallback(params);
  }
  
  // Generate summary for each chunk
  const partialSummaries: string[] = [];
  for (const chunk of splits) {
    partialSummaries.push(
      await summarizeWithFallback({
        ...params,
        messages: chunk,
        previousSummary: undefined,
      })
    );
  }
  
  if (partialSummaries.length === 1) return partialSummaries[0];
  
  // Merge all partial summaries
  const summaryMessages: AgentMessage[] = partialSummaries.map(
    (summary) => ({ role: "user", content: summary, timestamp: Date.now() })
  );
  
  const mergeInstructions = 
    "Merge these partial summaries into a single cohesive summary. " +
    "Preserve decisions, TODOs, open questions, and any constraints.";
  
  return summarizeWithFallback({
    ...params,
    messages: summaryMessages,
    customInstructions: mergeInstructions,
  });
}
```

Flow diagram:

```
Original messages [M1, M2, M3, ..., Mn]
           │
     splitMessagesByTokenShare(2)
           │
     ┌─────┴─────┐
     │           │
  Chunk A     Chunk B
  [M1..Mk]    [Mk+1..Mn]
     │           │
  summarize   summarize
     │           │
  Summary A   Summary B
     │           │
     └─────┬─────┘
           │
     Merge summaries request
     "Merge these partial summaries..."
           │
        Final summary
```

### Progressive Degradation (summarizeWithFallback)

If full summarization fails (e.g., messages too large), the system tries degradation strategies:

```typescript
export async function summarizeWithFallback(params: {
  messages: AgentMessage[];
  model: NonNullable<ExtensionContext["model"]>;
  // ...
}): Promise<string> {
  // Attempt 1: Full summarization
  try {
    return await summarizeChunks(params);
  } catch (fullError) {
    console.warn("Full summarization failed, trying partial...");
  }
  
  // Attempt 2: Summarize only small messages, skip oversized ones
  const smallMessages: AgentMessage[] = [];
  const oversizedNotes: string[] = [];
  
  for (const msg of messages) {
    if (isOversizedForSummary(msg, contextWindow)) {
      const tokens = estimateTokens(msg);
      oversizedNotes.push(
        `[Large ${role} (~${Math.round(tokens/1000)}K tokens) ` +
        `omitted from summary]`
      );
    } else {
      smallMessages.push(msg);
    }
  }
  
  if (smallMessages.length > 0) {
    try {
      const partialSummary = await summarizeChunks({
        ...params, messages: smallMessages
      });
      const notes = oversizedNotes.join("\n");
      return partialSummary + (notes ? `\n\n${notes}` : "");
    } catch {
      console.warn("Partial summarization also failed.");
    }
  }
  
  // Final degradation: only record message count
  return `Context contained ${messages.length} messages ` +
    `(${oversizedNotes.length} oversized). ` +
    `Summary unavailable due to size limits.`;
}
```

Three-level degradation strategy:

| Level | Strategy | When Triggered |
|-------|----------|----------------|
| 1 | Full summarization | Default attempt |
| 2 | Partial summarization (skip oversized messages) | Full summarization fails |
| 3 | Statistics only | Partial summarization also fails |

## 9.4.3 Memory Flush Before Compaction (Memory Flush)

Before compaction, all unsaved tool call results must be written to the session record. This is because compaction **discards** original messages, and if tool results aren't saved, this information is lost forever.

The `compactEmbeddedPiSessionDirect` function in `compact.ts` performs flushing after compaction:

```typescript
// src/agents/pi-embedded-runner/compact.ts (simplified)
export async function compactEmbeddedPiSessionDirect(
  params: CompactEmbeddedPiSessionParams,
): Promise<EmbeddedPiCompactResult> {
  // ...preparation work: model parsing, API key retrieval, etc.
  
  const sessionLock = await acquireSessionWriteLock({
    sessionFile: params.sessionFile,
  });
  
  try {
    // Repair potentially corrupted session file
    await repairSessionFileIfNeeded({ sessionFile: params.sessionFile });
    
    // Create Agent session
    const { session } = await createAgentSession({ ... });
    
    // Apply system prompt override
    applySystemPromptOverrideToSession(session, systemPromptOverride());
    
    try {
      // Clean and validate conversation history
      const sanitized = await sanitizeSessionHistory({ ... });
      const validated = validateAnthropicTurns(
        validateGeminiTurns(sanitized)
      );
      const limited = limitHistoryTurns(validated, historyLimit);
      
      if (limited.length > 0) {
        session.agent.replaceMessages(limited);
      }
      
      // Perform compaction
      const result = await session.compact(params.customInstructions);
      
      // Estimate tokens after compaction
      let tokensAfter: number | undefined;
      try {
        tokensAfter = 0;
        for (const message of session.messages) {
          tokensAfter += estimateTokens(message);
        }
        if (tokensAfter > result.tokensBefore) {
          tokensAfter = undefined; // Estimate unreliable
        }
      } catch {
        tokensAfter = undefined;
      }
      
      return {
        ok: true,
        compacted: true,
        result: {
          summary: result.summary,
          firstKeptEntryId: result.firstKeptEntryId,
          tokensBefore: result.tokensBefore,
          tokensAfter,
        },
      };
    } finally {
      // Critical: flush pending tool results
      sessionManager.flushPendingToolResults?.();
      session.dispose();
    }
  } finally {
    await sessionLock.release();
  }
}
```

### Session Write Lock

Compaction operations need exclusive access to the session file, implemented via `acquireSessionWriteLock`:

```typescript
const sessionLock = await acquireSessionWriteLock({
  sessionFile: params.sessionFile,
});
```

> **Sidebar — Write Lock**
>
> A write lock is a concurrency control mechanism: when an operation holds a write lock, other operations cannot simultaneously read or write the same resource. OpenClaw acquires a write lock during compaction, preventing other threads (like user message processing) from simultaneously modifying the session file, thus avoiding data races and file corruption.

### History Cleanup Flow

Before compaction, OpenClaw performs a series of cleanup operations on conversation history:

```
Original messages → sanitizeSessionHistory → validateGeminiTurns 
  → validateAnthropicTurns → limitHistoryTurns → Compaction
```

1. **sanitizeSessionHistory**: Remove incomplete messages, clean image data, etc.
2. **validateGeminiTurns**: Ensure Gemini model's required "user first" order
3. **validateAnthropicTurns**: Ensure Anthropic model's user/assistant alternation
4. **limitHistoryTurns**: Limit maximum conversation turns per configuration

### Lane Queuing

Compaction can be called in two contexts:

```typescript
// Method 1: Direct call (already inside Lane)
export async function compactEmbeddedPiSessionDirect(params) { ... }

// Method 2: Queue through Lane (external call)
export async function compactEmbeddedPiSession(params) {
  const sessionLane = resolveSessionLane(params.sessionKey || params.sessionId);
  const globalLane = resolveGlobalLane(params.lane);
  return enqueueCommandInLane(sessionLane, () =>
    enqueueGlobal(async () => compactEmbeddedPiSessionDirect(params))
  );
}
```

`compactEmbeddedPiSession` ensures through **dual Lane queuing** (session Lane + global Lane) that compaction operations don't conflict with other operations on the same session and don't exceed global concurrency limits. `compactEmbeddedPiSessionDirect` is used for scenarios already inside a Lane (like auto-compaction retries), avoiding deadlocks.

## 9.4.4 Compaction Retry and Buffer Reset

### Safeguard Mode: Advanced Compaction Strategy

The "safeguard" mode implements smarter compaction through the `compaction-safeguard` extension:

```typescript
// src/agents/pi-extensions/compaction-safeguard.ts (simplified)
export default function compactionSafeguardExtension(api: ExtensionAPI): void {
  api.on("session_before_compact", async (event, ctx) => {
    const { preparation, customInstructions, signal } = event;
    const { readFiles, modifiedFiles } = computeFileLists(preparation.fileOps);
    const toolFailures = collectToolFailures([
      ...preparation.messagesToSummarize,
      ...preparation.turnPrefixMessages,
    ]);
    
    // Build fallback summary (used when no model available)
    const fallbackSummary = 
      `${FALLBACK_SUMMARY}${toolFailureSection}${fileOpsSummary}`;
    
    const model = ctx.model;
    const apiKey = await ctx.modelRegistry.getApiKey(model);
    if (!model || !apiKey) {
      return { compaction: { summary: fallbackSummary, ... } };
    }
    
    // Calculate context budget
    const contextWindowTokens = runtime?.contextWindowTokens ?? modelContextWindow;
    const maxHistoryShare = runtime?.maxHistoryShare ?? 0.5;
    
    // If new content (system prompt + tool Schema) uses too much space,
    // first prune history messages
    if (newContentTokens > maxHistoryTokens) {
      const pruned = pruneHistoryForContextShare({
        messages: messagesToSummarize,
        maxContextTokens: contextWindowTokens,
        maxHistoryShare,
        parts: 2,
      });
      
      if (pruned.droppedChunks > 0) {
        // Also generate summary for pruned messages
        droppedSummary = await summarizeInStages({
          messages: pruned.droppedMessagesList,
          ...
        });
        messagesToSummarize = pruned.messages;
      }
    }
    
    // Main summary: adaptive chunking
    const adaptiveRatio = computeAdaptiveChunkRatio(
      allMessages, contextWindowTokens
    );
    const historySummary = await summarizeInStages({
      messages: messagesToSummarize,
      maxChunkTokens: Math.floor(contextWindowTokens * adaptiveRatio),
      previousSummary: droppedSummary, // Include pruned message summary
      ...
    });
    
    // Append tool failure info and file operation records
    summary += toolFailureSection;
    summary += fileOpsSummary;
    
    return { compaction: { summary, ... } };
  });
}
```

Core improvements in Safeguard mode:

### 1. History Pruning (History Pruning)

When system prompt + tool Schema already consume significant context, space for conversation history may be insufficient. `pruneHistoryForContextShare` iteratively prunes the oldest message chunks:

```typescript
export function pruneHistoryForContextShare(params: {
  messages: AgentMessage[];
  maxContextTokens: number;
  maxHistoryShare?: number; // Default 0.5 (50%)
  parts?: number;
}): {
  messages: AgentMessage[];        // Retained messages
  droppedMessagesList: AgentMessage[]; // Pruned messages
  droppedChunks: number;
  droppedMessages: number;
  droppedTokens: number;
  keptTokens: number;
  budgetTokens: number;
} {
  const budgetTokens = Math.floor(maxContextTokens * maxHistoryShare);
  let keptMessages = params.messages;
  
  while (keptMessages.length > 0 && 
         estimateMessagesTokens(keptMessages) > budgetTokens) {
    const chunks = splitMessagesByTokenShare(keptMessages, parts);
    if (chunks.length <= 1) break;
    
    const [dropped, ...rest] = chunks;
    const flatRest = rest.flat();
    
    // Repair tool call pairing
    const repairReport = repairToolUseResultPairing(flatRest);
    keptMessages = repairReport.messages;
    
    allDroppedMessages.push(...dropped);
    droppedChunks++;
  }
  
  return { messages: keptMessages, droppedMessagesList: allDroppedMessages, ... };
}
```

> **Sidebar — Tool Call Pairing Repair**
>
> In Anthropic's API, each `tool_use` (tool call) must have a corresponding `tool_result` (tool result), and vice versa. When pruning history messages, "orphan" `tool_result` messages may appear — their corresponding `tool_use` is in a pruned chunk. `repairToolUseResultPairing` detects and removes these orphans, preventing API errors like "unexpected tool_use_id".

### 2. Tool Failure Preservation

Compaction not only preserves conversation summaries but also specially preserves **tool execution failure** information:

```typescript
function collectToolFailures(
  messages: AgentMessage[]
): ToolFailure[] {
  const failures: ToolFailure[] = [];
  for (const message of messages) {
    if (role !== "toolResult" || toolResult.isError !== true) continue;

      meta: formatToolFailureMeta(toolResult.details),

  }
  return failures;
}
```

```
- exec (exitCode=1): npm test failed with 3 errors
- write: Permission denied: /etc/hosts
```

### 3. File Operation Records

The compaction summary also includes a list of files read and modified during the session:

```typescript
function computeFileLists(fileOps: FileOperations): {
  readFiles: string[];
  modifiedFiles: string[];
} {
  const modified = new Set([...fileOps.edited, ...fileOps.written]);
  const readFiles = [...fileOps.read]
    .filter((f) => !modified.has(f)).toSorted();
  const modifiedFiles = [...modified].toSorted();
  return { readFiles, modifiedFiles };
}
```

```xml
src/config/config.ts
</read-files>

<modified-files>

</modified-files>
```

### Compaction Event Subscription and Buffer Reset

From a user visibility perspective, message stream pushing needs to pause during compaction and reset the buffer after completion:

```typescript
// src/agents/pi-embedded-subscribe.handlers.lifecycle.ts (simplified)
case "auto_compaction_start":
  ctx.state.compactionInFlight = true;
  ctx.emit({ stream: "compaction", type: "start" });
  break;

case "auto_compaction_end":
  ctx.state.compactionInFlight = false;
  if (event.willRetry) {
    // Compaction complete but will retry → wait for next round
    ctx.emit({ stream: "compaction", type: "retry" });
  } else {
    // Compaction completely done → unblock
    ctx.emit({ stream: "compaction", type: "end" });
  }
  break;
```

The subscription side waits for compaction completion via `compactionRetryPromise`:

```typescript
// src/agents/pi-embedded-subscribe.ts (simplified)
isCompacting: () => 
  state.compactionInFlight || state.pendingCompactionRetry > 0,

waitForCompaction: () => {
  if (state.compactionInFlight || state.pendingCompactionRetry > 0) {
    return state.compactionRetryPromise ?? Promise.resolve();
  }
}
```

This ensures:
- During compaction, the frontend doesn't receive incomplete text streams
- During compaction retry, the buffer is properly reset
- After compaction completes, new text streams start from a clean state

## Section Summary

1. **Context compaction** is the core mechanism enabling OpenClaw's "infinite conversations" — by summarizing old messages to free up context window space.
2. **Trigger strategy** is passive: compaction starts only when the model API returns context overflow error, with maximum 3 retries; compaction failure itself doesn't trigger further compaction.
3. **Staged summarization** (summarizeInStages) chunks messages, summarizes each chunk separately, then merges into final summary, effectively handling oversized contexts.
4. **Adaptive chunking** (computeAdaptiveChunkRatio) dynamically adjusts chunk ratio based on average message size, avoiding exceeding model limits.
5. **Three-level degradation** strategy (full summarization → partial summarization → statistics) ensures compaction doesn't completely fail due to a few oversized messages.
6. **Safeguard mode** adds enhancements like history pruning, tool failure preservation, and file operation records, generating more informative compaction summaries.
7. **Concurrency safety** is guaranteed through write locks and Lane queuing, ensuring session files aren't concurrently modified during compaction.
