## 25.2.1 `gateway` ŌĆö Start/Manage Gateway Server

`gateway` is OpenClaw's most critical commandŌĆöit manages the startup, shutdown, and administration of the WebSocket Gateway server.

### Subcommand Overview

| Subcommand | Description |
|------------|-------------|
| `gateway` (default) | Run Gateway server in foreground |
| `gateway run` | Explicit foreground run |
| `gateway status` | Display service status + probe Gateway |
| `gateway install` | Install system service (launchd/systemd/schtasks) |
| `gateway uninstall` | Uninstall system service |
| `gateway start` | Start system service |
| `gateway stop` | Stop system service |
| `gateway restart` | Restart system service |
| `gateway call <method>` | Call Gateway RPC method |
| `gateway health` | Get Gateway health status |
| `gateway probe` | Comprehensive probe (reachability + discovery + health) |
| `gateway discover` | Discover Gateway instances on LAN via Bonjour |
| `gateway usage-cost` | Query usage cost statistics |

### Service Lifecycle Management

The five commands `gateway install/uninstall/start/stop/restart` delegate to `daemon-cli.ts`, selecting different service management backends based on OS:

- **macOS** ŌĆö launchd (`launchctl`)
- **Linux** ŌĆö systemd (`systemctl`)
- **Windows** ŌĆö schtasks (Task Scheduler)

```bash
# Install as system service, specify port and runtime
openclaw gateway install --port 18789 --runtime node

# Override existing service configuration with --force
openclaw gateway install --force

# View service status + probe running Gateway
openclaw gateway status --deep
```

### Gateway Discovery (Bonjour)

`gateway discover` uses mDNS/Bonjour protocol to search for running Gateway instances on the LAN:

```typescript
// src/cli/gateway-cli/register.ts (simplified)
const beacons = await discoverGatewayBeacons({ timeoutMs, wideAreaDomain });
const deduped = dedupeBeacons(beacons); // Deduplicate
```

> **Sidebar ŌĆö Bonjour/mDNS**
>
> Bonjour (also called mDNS/DNS-SD) is a zero-configuration network protocol developed by Apple that allows devices to automatically discover services on a LAN without a DNS server. When Gateway starts, it broadcasts its presence to the LAN (including IP address and port), and other devices can find it via Bonjour. This enables multiple devices to automatically discover and connect to Gateway on home/office networks.

### Gateway RPC Invocation

`gateway call` provides a generic RPC invocation interface, allowing direct calls to Gateway-exposed methods:

```bash
# Call health method
openclaw gateway call health --json

# Call cron.list with parameters
openclaw gateway call cron.list --params '{"active": true}'
```

## 25.2.2 `agent` ŌĆö Send Message to AI Agent

The `agent` command is the primary way users directly converse with the AI Agent, sending messages through the Gateway WebSocket connection and receiving replies.

```bash
# Basic usage: start new session with specified number
openclaw agent --to +15555550123 --message "Analyze today's logs"

# Specify Agent and thinking level
openclaw agent --agent ops --message "Summarize report" --thinking medium

# Continue existing session
openclaw agent --session-id 1234 --message "Continue the analysis above"

# Let Agent reply and send via Slack
openclaw agent --message "Generate weekly report" --deliver --reply-channel slack --reply-to "#reports"
```

### Key Options

| Option | Description |
|--------|-------------|
| `--message <text>` | **Required**, message content |
| `--to <number>` | Recipient number (E.164 format), used to derive session key |
| `--session-id <id>` | Directly specify session ID |
| `--agent <id>` | Specify Agent (override routing binding) |
| `--thinking <level>` | Thinking level: off / minimal / low / medium / high |
| `--channel <channel>` | Delivery channel: last / whatsapp / telegram / discord / ... |
| `--deliver` | Send Agent reply to selected channel |
| `--local` | Run embedded Agent locally (requires API key) |
| `--json` | Output result in JSON format |
| `--timeout <seconds>` | Timeout (default 600 seconds) |

### agents Management Subcommands

Besides `agent` (singular), there's an `agents` (plural) command for managing multi-Agent configuration:

```bash
openclaw agents list --bindings    # List all Agents and routing bindings
openclaw agents add ops --workspace ~/ops --model gpt-4  # Add new Agent
openclaw agents set-identity --agent main --from-identity  # Set identity from IDENTITY.md
openclaw agents delete ops --force   # Delete Agent
```

## 25.2.3 `send` ŌĆö Send Message via Channel

The `send` command (actually registered as `message send`) allows sending messages directly through a specified channel, bypassing the Agent layer:

```bash
openclaw message send --target +15555550123 --message "Hi"
openclaw message send --channel telegram --target @mychat --message "Hello"
openclaw message send --target +15555550123 --message "Check this photo" --media photo.jpg
```

The `message` command group includes rich subcommands:

| Subcommand | Description |
|------------|-------------|
| `message send` | Send text/media message |
| `message broadcast` | Broadcast message |
| `message poll` | Create poll |
| `message react` / `unreact` | Add/remove emoji reaction |
| `message read` / `edit` / `delete` | Read/edit/delete message |
| `message pin` / `unpin` | Pin/unpin message |
| `message permissions` | Permission management |
| `message search` | Search messages |
| `message thread` | Thread message operations |
| `message emoji` / `sticker` | Emoji and sticker management |

Each subcommand's registration is split into independent module files (e.g., `register.send.ts`, `register.poll.ts`), sharing channel options and common logic through `createMessageCliHelpers`.

## 25.2.11 `doctor` ŌĆö Health Check and Diagnostics

`doctor` is OpenClaw's "general practitioner"ŌĆöit detects and fixes common issues:


The `doctor` command is registered in `register.maintenance.ts`, grouped with `dashboard`, `reset`, and `uninstall` commands as maintenance commands:

| Command | Description |
|---------|-------------|
| `doctor` | Health check + quick fix |
| `dashboard` | Open Web console |
| `reset` | Reset local config/state |
| `uninstall` | Uninstall Gateway service + local data |

The special aspect of `doctor`ŌĆöit's whitelisted in the config guard (`ALLOWED_INVALID_COMMANDS`), allowed to run even if the config file is invalid, since it's designed to fix config issues.

## 25.2.12 `security` ŌĆö Security Audit

```bash

openclaw security audit --deep    # Include Gateway online probing
openclaw security audit --fix     # Apply security fixes
```

The `security audit` implementation has two steps:

1. **Fix** (optional): `fixSecurityFootguns()` tightens default config, corrects file permissions (chmod)
2. **Audit**: `runSecurityAudit()` five-layer scan (attack surface ŌåÆ config ŌåÆ filesystem ŌåÆ channels ŌåÆ deep probing)

Output is grouped by severity (CRITICAL ŌåÆ WARN ŌåÆ INFO), each finding includes `checkId`, title, details, and fix suggestions.

## 25.2.14 `update` ŌĆö Version Update

The `update` command manages OpenClaw version upgrades, supporting three update channels:

```bash
openclaw update                    # Update using current channel
openclaw update --channel beta     # Switch to Beta channel
openclaw update --channel dev      # Switch to Dev channel (Git)
openclaw update status             # View update status
openclaw update wizard             # Interactive update wizard
openclaw --update                  # Shorthand form
```

### Three Update Channels

| Channel | Install Method | Source |
|---------|----------------|--------|
| `stable` | npm (`latest` tag) | Official release versions |
| `beta` | npm (`beta` tag) | Pre-release versions |
| `dev` | Git (main branch) | Source code compilation |

### Update Flow

The `update` command implements a complete update orchestration flow:

1. **Detect install type**: Git source vs npm global package
2. **Resolve target version**: Get version number from npm registry tag
3. **Downgrade confirmation**: If target version is lower than current, require user confirmation
4. **Channel switch**: If new channel specified, write to config file
5. **Execute update**:
   - **Git mode**: fetch ŌåÆ rebase ŌåÆ install deps ŌåÆ build ŌåÆ doctor
   - **npm mode**: Detect package manager (npm/pnpm/bun) ŌåÆ global install
   - **Channel switch**: `dev` ŌåÆ clone Git repo; non-`dev` ŌåÆ global npm install
6. **Update plugins**: Sync built-in plugins, update npm-installed plugins
7. **Shell completion**: Update cache, prompt for installation
8. **Restart service**: Restart Gateway daemon + run doctor

The entire flow includes progress indicators (spinner), each step showing elapsed time and success/failure status. After update completes, a random fun quote is output (e.g., "The lobster has molted. Harder shell, sharper claws.").

## Section Summary

1. **`gateway`** is the most critical command, managing the complete lifecycle of WebSocket Gateway (foreground run/system service/probe/discovery/RPC invocation), with cross-platform support for launchd/systemd/schtasks
2. **`agent`/`message`** handle AI conversation and raw message sending respectivelyŌĆö`agent` routes through Gateway to AI Agent, `message` directly operates channels
3. **`channels`/`models`** manage two major external dependenciesŌĆöchat channels and AI models, where `models` command tree reaches three levels deep, including complete auth, alias, and fallback chain management
4. **`doctor`/`security`/`onboard`** form the operations toolkitŌĆödiagnose/repair, security audit, guided installation; `doctor` is specially exempted from config validation
5. **`update`** implements three-channel (stable/beta/dev) update orchestration, auto-detecting install type, managing package managers, syncing plugins, restarting services
6. **All commands uniformly use** `runCommandWithRuntime(defaultRuntime, action)` wrapper, ensuring consistent error handling; support `--json` machine-readable output mode
