OpenClaw is a **personal AI assistant** that runs on your own device. Unlike cloud-based services like ChatGPT or Claude.ai, OpenClaw's core philosophy is: **your AI assistant should belong to you**—running on your machine, connected to the messaging platforms you already use, with all data stored on your local disk.

As the OpenClaw project's README states:

> *"If you want a personal, single-user assistant that feels local, fast, and always-on, this is it."*

## 1.1.1 The "Local-first" Philosophy and Comparison with Traditional SaaS

Over the past few years, the "Local-first" philosophy has gradually gained traction in software engineering. This concept was systematically articulated by Martin Kleppmann and others in their 2019 paper *"Local-first software: you own your data, in spite of the cloud"*. Its core principles are:

1. **Data Ownership**: User data should be stored on devices controlled by the user, not on third-party servers
2. **Offline Availability**: Software should function without network connectivity (at least partially)
3. **Low Latency**: Local operations don't require network round-trips, resulting in faster responses
4. **Privacy Protection**: Sensitive data doesn't need to leave the user's device

OpenClaw embodies these principles. The following table contrasts OpenClaw with typical cloud-based AI assistants:

| Dimension | Cloud AI Assistant (e.g., ChatGPT) | OpenClaw |
|-----------|-----------------------------------|----------|
| Execution Location | Service provider's cloud servers | Your own device (laptop, server, VPS) |
| Data Storage | Provider's database | Local file system (`~/.openclaw/`) |
| Conversation History | Provider-controlled | Local JSONL files, directly readable by you |
| Memory Mechanism | Provider's proprietary implementation | Pure Markdown files (`MEMORY.md`) |
| Message Entry Point | Provider's web/app | Your existing messaging platforms (WhatsApp, Telegram, etc.) |
| Customization Capability | Limited GPTs/Gems | Fully programmable skill system + source-level customization |
| Cost Model | Monthly subscription | Pay-per-API-call (or subscription OAuth) |

Of course, OpenClaw still needs to call cloud-based large language model APIs (such as Anthropic Claude or OpenAI GPT) for inference—local devices typically lack the computational power to run large models. But the key difference is: **the control plane is local**. Your configuration, conversation records, workspace files, and memory—all of these are on your disk, and you can view, backup, or migrate them at any time.

From a source code perspective, OpenClaw's data directory structure clearly reflects this design philosophy:

```
~/.openclaw/
├── openclaw.json          # Main configuration file (JSON5 format)
├── credentials/           # Channel credentials (WhatsApp session, etc.)
├── workspace/             # Workspace root directory
│   ├── AGENTS.md          # Agent behavior instructions
│   ├── SOUL.md            # Agent personality definition
│   ├── TOOLS.md           # Tool usage instructions
│   ├── MEMORY.md          # Long-term memory
│   ├── memory/            # Daily memory logs
│   │   └── 2026-02-17.md
│   └── skills/            # Workspace skills
│       └── my-skill/
│           └── SKILL.md
└── agents/
    └── main/
        ├── sessions/      # Session storage
        │   ├── sessions.json  # Session metadata
        │   └── *.jsonl        # Conversation transcripts
        └── models.json    # Model configuration
```

## 1.1.2 Single-user Design: Why Not Multi-tenant?

OpenClaw has been designed as a **single-user** system from the start. This is an intentional design decision, not a capability limitation. In the `README.md`, the project describes itself this way:

> *"The Gateway is just the control plane — the product is the assistant."*

In multi-tenant systems, a single service instance serves multiple users simultaneously. This requires solving complex problems like data isolation, access control, and resource quotas. OpenClaw's single-user design brings several significant simplifications:

1. **Simplified Trust Model**: By default, tools in the main session have full access to the host machine—because the user is the machine's owner. As stated in the `README.md`'s security section:

   > *"Default: tools run on the host for the main session, so the agent has full access when it's just you."*

2. **Simplified Session Model**: All direct messages (DMs) share the same main session by default, providing conversation continuity across devices and channels. What you say to the assistant on WhatsApp can be continued on Telegram.

3. **Simplified Configuration**: A single JSON5 configuration file manages everything, with no need for user management or permission hierarchies.

4. **Simplified Deployment**: One machine runs one Gateway instance, with no need to consider horizontal scaling or load balancing.

Of course, this doesn't mean OpenClaw completely ignores multi-user scenarios. When your assistant is configured to receive messages from different people (for example, in groups or with an open DM whitelist), OpenClaw provides a **sandbox mechanism** to isolate non-main sessions:

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",  // Non-main sessions run in Docker sandbox
      },
    },
  },
}
```

Bash commands in non-main sessions execute in isolated Docker containers, ensuring that messages from different sources don't interfere with the host machine or each other. We'll discuss the security model in detail in Chapter 24.

## 1.1.3 Multi-channel Unified Inbox

One of OpenClaw's most striking features is its **multi-channel unified inbox**. You don't need to deploy a separate bot for each messaging platform—a single OpenClaw instance can simultaneously connect to all supported platforms:

**Core Channels** (built into `src/`):

| Channel | Underlying Library | Source Location |
|---------|-------------------|-----------------|
| WhatsApp | `@whiskeysockets/baileys` | `src/whatsapp/`, `src/web/` |
| Telegram | `grammy` | `src/telegram/` |
| Discord | `@buape/carbon` + discord.js | `src/discord/` |
| Slack | `@slack/bolt` | `src/slack/` |
| Signal | `signal-cli` | `src/signal/` |
| iMessage (legacy) | Native macOS `imsg` | `src/imessage/` |
| WebChat | Built into Gateway | `src/channels/web/` |

**Extension Channels** (located in `extensions/`, optional installation):

| Channel | Source Location |
|---------|-----------------|
| Microsoft Teams | `extensions/msteams/` |
| BlueBubbles (iMessage) | `extensions/bluebubbles/` |
| Matrix | `extensions/matrix/` |
| Google Chat | `extensions/googlechat/` |
| Nostr | `extensions/nostr/` |
| Twitch | `extensions/twitch/` |
| Zalo / Zalo Personal | `extensions/zalo/`, `extensions/zalouser/` |
| LINE | `extensions/line/` |
| Feishu (Lark) | `extensions/feishu/` |
| Mattermost | `extensions/mattermost/` |

The core idea behind this design is: **messaging channels are just the transport layer; the AI assistant is the product**. Regardless of which channel a message comes from, it's normalized into a unified internal format, routed to the same AI Agent for processing, and the response is sent back through the corresponding channel.

From the user's perspective, this means you can:

- Ask the assistant to check your schedule on WhatsApp in the morning
- Have it summarize documents in a Slack work channel in the afternoon
- Ask it to record your thoughts on Telegram in the evening
- And these conversations share the same context—the assistant "remembers" what you said on different platforms

From an engineering perspective, the key to implementing this capability is OpenClaw's **Channel Adapter** architecture. Each channel implements a unified interface responsible for:

1. **Inbound**: Listening to channel messages and converting them to OpenClaw's internal format
2. **Outbound**: Receiving OpenClaw's responses and converting them to channel-specific format for sending
3. **Capability Declaration**: Informing the Gateway which features the channel supports (images, audio, Markdown, reactions, threads, etc.)

We'll dive deep into the implementation of this adapter architecture in Chapters 11-13.
