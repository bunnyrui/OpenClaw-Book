Before diving into the source code, we need to have a comprehensive understanding of OpenClaw's project directory structure. A good directory structure is like a book's table of contents—it reflects the project's architectural decisions and module divisions.

The top-level structure of the OpenClaw repository root directory is as follows:

```
openclaw/
├── src/                    # Core source code (TypeScript)
├── apps/                   # Native client applications
│   ├── macos/             # macOS menu bar application (Swift)
│   ├── ios/               # iOS node application (Swift)
│   ├── android/           # Android node application (Kotlin)
│   └── shared/            # Cross-platform shared code (Swift)
├── ui/                     # Web console UI (Lit + Vite)
├── extensions/             # Optional channel extensions (31)
├── packages/               # Internal shared packages
├── skills/                 # Built-in skills (52)
├── docs/                   # Official documentation source
├── scripts/                # Build and tool scripts
├── test/                   # Global test configuration
├── vendor/                 # Third-party code
├── patches/                # pnpm patch patches
├── git-hooks/              # Git hooks
├── package.json            # Main package configuration
├── pnpm-workspace.yaml     # Monorepo workspace definition
```

Let's dive into each core directory.

## 1.4.1 `src/` — Core Source Code (69 Subdirectories)

`src/` is OpenClaw's core, containing TypeScript source code for all core functionality including Gateway, Agent, channels, tools, and more. It has 69 subdirectories—the largest part of the entire project.

Divided by functional domain, these subdirectories can be grouped as follows:

### Gateway Core (Control Plane)

| Directory | Files | Description |
|-----------|-------|-------------|
| `src/gateway/` | 128 | Gateway WebSocket server, HTTP server, session management, method handling |
| `src/routing/` | 5 | Message routing logic: route resolution, session key generation |
| `src/sessions/` | 7 | Session helpers: level overrides, model overrides, send policies |

### Agent Runtime

| Directory | Files | Description |
|-----------|-------|-------------|
| `src/agents/` | 308 | **Largest directory**. Agent loop, tool system, model management, skills, sandbox |
| `src/providers/` | 8 | Model provider-specific logic (GitHub Copilot, Google, Qwen) |

## 1.4.2 `apps/` — Native Client Applications

```

│   │   ├── Views/         # SwiftUI views
│   │   ├── VoiceWake/     # Voice wake module
│   │   └── ...
│   ├── project.yml        # XcodeGen project description

│   ├── Sources/           # Swift source code

├── android/               # Android node application
│   ├── app/
│   │   └── src/main/
│   │       ├── java/ai/openclaw/android/  # Kotlin source code
│   │       ├── res/       # Android resources
│   │       └── AndroidManifest.xml
│   ├── build.gradle.kts   # Gradle build configuration

    └── OpenClawKit/
        └── Sources/       # Shared Swift Package
```

The macOS and iOS applications use **XcodeGen** to generate Xcode project files (rather than directly committing `.xcodeproj`), which avoids merge conflict issues with Xcode project files.

## 1.4.3 `ui/` — Web Console UI

```
ui/
├── index.html             # SPA entry point
├── package.json           # UI package configuration (Lit + Vite)
├── vite.config.ts         # Vite build configuration
├── vitest.config.ts       # UI test configuration
├── public/                # Static assets
└── src/                   # Lit Web Components source code
```

The UI is an independent pnpm workspace package (`openclaw-control-ui`) built with Vite. The build artifacts are embedded in Gateway and served directly, requiring no separate deployment. Gateway automatically provides UI service on startup (`src/gateway/control-ui.ts`).
